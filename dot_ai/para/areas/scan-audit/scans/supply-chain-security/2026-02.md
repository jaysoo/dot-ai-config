# Supply Chain Security Review: Nx Platform (2026-02)

**Status**: Clear (with recommendations)
**Date**: 2026-02-27
**Lookback**: 2025-12-29 to 2026-02-27
**Reviewer**: Automated scan + manual analysis

## Summary

No critical supply chain issues found. All Nx packages publish from a well-secured GitHub Actions workflow with SHA-pinned actions, SLSA provenance attestations, environment protection (`npm-registry`), `id-token: write` for OIDC-based npm publishing, and a hardcoded allowlist of 4 GitHub users authorized to publish to `latest`. One documentation-only commit touched the publish workflow area during the lookback period. Several typosquat-adjacent package names exist on npm but pose minimal risk (old, unrelated, or unpublished).

## Action Required

- **None critical.** Recommendations below for hardening.

### Recommendations

1. **Add `provenance: true` to nx.json release config or .npmrc** -- Provenance attestations ARE being generated (confirmed via npm attestation API), likely via npm's automatic OIDC-based provenance when `id-token: write` is set. However, explicitly setting `provenance=true` in `.npmrc` or release config would make this intent clear and prevent accidental regression.
2. **Audit the `nrwl` npm package name** -- The `nrwl` package was published once (v213.21.24 on 2024-06-10) then unpublished. Since `nrwl` is the org name, consider re-claiming this package name as a placeholder to prevent future squatting.
3. **Consider reducing maintainer count on `nx` core package** -- The `nx` package has 8 maintainers. The `@nx/*` scoped packages have 5. Fewer maintainers reduces attack surface.
