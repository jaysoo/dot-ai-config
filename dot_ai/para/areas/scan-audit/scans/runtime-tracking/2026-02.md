## Action Items

### HIGH Priority

1. **Node.js 20 EOL ‚Äî April 30, 2026**
   - Node 20 reaches End of Life in ~62 days
   - Nx should update minimum supported Node version for the next major release
   - Node 22 LTS (Jolt) is the recommended migration target (EOL April 2027)
   - Node 24 LTS (Krypton) is active LTS (EOL April 2028)
   - Action: Plan blog post, migration guide, minimum version bump in Nx 23+

2. **January 13, 2026 Security Release ‚Äî Patch All Active Lines**
   - 6 CVEs fixed across Node 20.20.0, 22.22.0, 24.13.0, 25.3.0
   - CVE-2025-55131: Unsafe buffer creation (zero-fill toggle removed)
   - CVE-2025-59465: TLSSocket default error handler
   - CVE-2025-55130: Symlink permission bypass
   - CVE-2025-55132: futimes permission model bypass
   - CVE-2025-59466: Stack overflow in async_hooks
   - CVE-2026-21637: TLS callback exception routing
   - Action: Recommend users update to patched versions in Nx docs/blog

3. **Bun bun.lock Parsing Broken for Nx (ongoing)**
   - Bun's lockfile format has parsing issues with Nx
   - Related: https://github.com/oven-sh/bun/issues/16252
   - Affects @nx/js lockfile parsing, `nx release` publish, dependency graph
   - Action: Track Bun lockfile stabilization, consider dedicated parser

### MEDIUM Priority

4. **Node 25 Current Line ‚Äî Breaking Changes to Monitor**
   - Corepack removed from Node.js distribution
   - `SlowBuffer` removed, `_stream_*` modules deprecated
   - `rmdir` recursive removed (use `rm` with `recursive: true`)
   - `fs.F_OK/R_OK/W_OK/X_OK` removed from fs module
   - `child_process._channel` removed
   - Action: Audit Nx codebase for deprecated API usage before Node 25 becomes Node 26 LTS candidate

5. **Deno 2.7 npm Overrides Support**
   - Deno now supports `overrides` in package.json
   - Improved `node:worker_threads` and `node:child_process` compatibility
   - Action: Monitor Deno+Nx integration requests as compatibility grows

### LOW Priority

6. **Node 24.14.0 ‚Äî `#/` Subpath Imports**
   - New subpath imports starting with `#/` syntax
   - `fs.watch` ignore option useful for file watchers
   - `http.setGlobalProxyFromEnv()` new utility
   - Action: Consider adopting `#/` imports in Nx internal code

7. **Bun 1.3.10 ‚Äî TC39 Decorators + Barrel Optimization**
   - Standard ES decorators now supported
   - Barrel import optimization improves monorepo performance
   - Action: No immediate action; monitor for Nx plugin compatibility

### Node 25.0.0 Breaking Changes (Released 2025-10-15)

These will matter when Node 26 goes LTS. Key items for Nx:

- **V8 upgraded to 14.1** ‚Äî Major `JSON.stringify` perf improvements, Uint8Array base64/hex
- **Corepack removed** ‚Äî No longer shipped with Node.js; users must install separately
- **`SlowBuffer` removed** ‚Äî Use `Buffer.allocUnsafe()` instead
- **`_stream_*` modules deprecated** ‚Äî Use `stream` module directly
- **`fs.F_OK/R_OK/W_OK/X_OK` removed** ‚Äî Use `fs.constants` instead
- **`rmdir` recursive removed** ‚Äî Use `rm({ recursive: true })` instead
- **`child_process._channel` removed** ‚Äî Internal API removed
- **`Module._debug` EOL** ‚Äî Internal debug function removed
- **NODE_MODULE_VERSION: 141** (was 138 in Node 24)
- **`--allow-net` permission model** ‚Äî New network access control
- **Web Storage enabled by default** ‚Äî No longer experimental
- **`ErrorEvent` now global** ‚Äî Exposed globally

### Nx Compatibility Risks

1. **`fs.F_OK` etc. removal in Node 25**: Audit Nx code for `fs.F_OK`, `fs.R_OK`, `fs.W_OK`, `fs.X_OK` usage ‚Äî should use `fs.constants.*` instead
2. **`_stream_*` deprecation**: Check if any Nx code imports `_stream_readable`, `_stream_writable`, etc.
3. **Corepack removal**: Nx users relying on Corepack for package manager management need guidance
4. **`rmdir` recursive removal**: Ensure Nx uses `rm` with `recursive: true` everywhere

### Nx Opportunities

1. **`fs.watch` ignore option (Node 24.14.0)**: Could improve Nx daemon file watching efficiency
2. **`#/` subpath imports (Node 24.14.0)**: Cleaner internal imports for Nx packages
3. **`http.setGlobalProxyFromEnv()` (Node 24.14.0)**: Useful for Nx Cloud proxy support
4. **SEA ESM support (Node 25.7.0)**: Future potential for Nx binary distribution
5. **SQLite moving to RC (Node 25.7.0)**: Could be used for Nx cache metadata

### Nx `engines` Field Status

- **`nx@22.5.3`**: No `engines` field set in package.json
- **`create-nx-workspace@22.5.3`**: No `engines` field set
- Nx does not enforce a minimum Node.js version at the package level
- **Recommendation**: Consider adding `"engines": { "node": ">=20.0.0" }` now, bump to `>=22.0.0` after Node 20 EOL

## TC39 Proposals

### Stage 4 (Shipping / Recently Shipped)

| Proposal | Impact on Nx |
|----------|-------------|
| **Explicit Resource Management** (`using`/`await using`) | High ‚Äî Nx daemon, file handles, DB connections. TypeScript already supports. |

### Stage 3 (Advancing)

| Proposal | Champions | Impact on Nx |
|----------|-----------|-------------|
| **Temporal** | Philipp Dunkel, Justin Grant et al. | Low ‚Äî Date/time API. Stable in Deno 2.7, Chrome 144. |
| **Decorators** | Kristen Hewell Garrett | Medium ‚Äî Already in TS. Bun 1.3.10 adds TC39 standard support. |
| **Decorator Metadata** | Kristen Hewell Garrett | Medium ‚Äî Complements decorators. |
| **Source Phase Imports** | Guy Bedford, Luca Casonato | Medium ‚Äî Module system evolution. WASM imports in Deno 2.6. |
| **Deferring Module Evaluation** | Yulia Startsev, Guy Bedford | High ‚Äî Lazy module loading. Could significantly improve Nx startup time. |
| **Atomics.pause** | Shu-yu Guo | Low ‚Äî Threading primitive. |
| **Joint Iteration** | Michael Ficarra | Low ‚Äî Iterator helpers. |

### Key TC39 Takeaway

**Deferring Module Evaluation** (Stage 3) is the most impactful proposal for Nx. It would allow lazy loading of plugin code, reducing startup time for `nx` CLI commands. Monitor closely.

### Deno 2.7 Highlights (2026-02-25)

- **Temporal API stabilized** ‚Äî First major runtime to ship stable Temporal
- **npm overrides support** ‚Äî `overrides` field in package.json now respected
- **`deno create`** ‚Äî Template scaffolding command
- **Windows ARM64** ‚Äî Platform expansion
- **`node:worker_threads` fixes** ‚Äî stdout/stdin forwarding, exit code propagation, cpuUsage()
- **`node:child_process` fixes** ‚Äî Socket-based stdio, timeout/killSignal, URL-based modulePath
- **Zstd compression** in `node:zlib`
- **`deno compile --self-extracting`** ‚Äî Enables native addon support in compiled binaries

### Deno 2.6 Highlights (2025-12-10)

- **`deno x` / `dx`** ‚Äî Package binary runner (like `npx`)
- **`deno audit`** ‚Äî Security vulnerability scanning
- **`@types/node` included by default** ‚Äî No manual setup needed
- **`#/` subpath imports** ‚Äî Same feature as Node 24.14.0
- **V8 14.2** ‚Äî Newer V8 than Node 25 (which has 14.1)
- **WASM source phase imports**
- **`--lockfile-only` flag** ‚Äî Update lockfile without downloading packages
- **BREAKING**: `deno install -g` requires `--` separator for script args

### Deno Deploy GA (2026-02-03)

- Deno Deploy is now Generally Available
- New "Deno Sandbox" API for running untrusted code in microVMs
- Potential competitor for Nx Cloud remote execution environments

### Nx Compatibility

- Deno's Node.js compatibility is rapidly improving (`worker_threads`, `child_process`, `zlib`)
- npm overrides support in 2.7 aligns Deno closer to Node.js package management
- `deno x` / `dx` provides npx-equivalent, making Nx CLI invocation possible
- Limited demand for Deno+Nx integration currently, but the gap is narrowing
