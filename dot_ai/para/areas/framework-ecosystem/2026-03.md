## Executive Summary

The JS ecosystem hit a major inflection point this period. Key developments since the last scan (Mar 17):

- **Next.js 16.2.0 STABLE released** (Mar 18) -- major performance release: ~87% faster dev startup, 25-60% faster SSR rendering (React contribution), Adapters API stable, `transitionTypes` for View Transitions, `unstable_catchError()` for granular error boundaries, 200+ Turbopack fixes. AI-focused features: `AGENTS.md` in create-next-app, browser log forwarding.
- **Vite 8.0.0 STABLE** (Mar 12) -- Rolldown-powered era is live. Compat layer auto-converts `rollupOptions`/`esbuildOptions`.
- **Rolldown at RC.9** (Mar 11) with weekly RC releases continuing. API frozen, powering Vite 8 stable.
- **Rspack 2.0 at beta.7** (Mar 17) -- weekly betas continuing toward stable. Breaking changes well-documented.
- **Angular 22.0.0-next.3** (Mar 12) -- selectorless components and signal-first APIs maturing. Angular 21.2.4 (Mar 12) is latest stable.
- **Nuxt 4.4.2** (Mar 12) -- vue-router v5 upgrade, custom useFetch factories, accessibility announcer.
- **Vue 3.6.0-beta.8** (Mar 16) -- Vapor Mode continues maturing.
- **React 19.2.4** -- no changes, security patches only.
- **TypeScript 6.0 beta** remains the biggest cross-cutting event.

**NEW since Mar 17**: Next.js 16.2.0 going stable is the most notable event -- performance improvements and Adapters API are now available for all users.

### Next.js 16 breaking changes (recap -- these are already stable)

**Removals:**

- Sync `params`/`searchParams`/`cookies()`/`headers()` access -- must use `await`
- `next lint` command removed -- use ESLint/Biome directly
- `experimental.ppr` and `experimental.dynamicIO` flags -- replaced by `cacheComponents: true`
- AMP support entirely removed
- `serverRuntimeConfig`/`publicRuntimeConfig` -- use env vars

**Behavior changes:**

- **Turbopack is default bundler** -- webpack via `--webpack` flag
- **`proxy.ts` replaces `middleware.ts`** -- Edge runtime not supported in proxy, Node.js runtime only
- **Parallel routes require explicit `default.js` files**
- **Node.js 20.9+ minimum** (Node 18 dropped)
- `revalidateTag()` requires cacheLife profile as 2nd argument

**New APIs:**

- Cache Components (`"use cache"` directive)
- `updateTag()` -- Server Actions API with read-your-writes semantics
- `refresh()` -- Server Actions API for uncached data refresh
- React Compiler support stable (`reactCompiler: true`, not default)

### Next.js 16.1 additions

- Turbopack File System Caching stable for dev
- Experimental Bundle Analyzer for Turbopack
- `next dev --inspect` debugging support

### Next.js 16.2 additions -- KEY EVENT (Mar 18)

**Performance:**

- **~87% faster `next dev` startup** (Time-to-URL)
- **25-60% faster SSR rendering** -- React contribution (`JSON.parse` reviver optimization, eliminates C++/JS boundary crossing)
- **2-20x faster `ImageResponse`** -- basic to complex images
- Default font changed from Noto Sans to Geist Sans in `ImageResponse`

**Stable features:**

- **Adapters API stable** -- platforms can customize build process (`adapterPath` config)
- **`transitionTypes` prop on `<Link>`** -- array of View Transition types for navigation animations (App Router only)
- **Server Function logging** in dev terminal (name, args, execution time, file)
- **Hydration diff indicator** -- clear server/client diff in error overlay (`+ Client` / `- Server`)
- **`next start --inspect`** -- attach Node.js debugger to production server
- **Error causes in dev overlay** -- `Error.cause` chain display up to 5 levels
- **Multiple icon formats** -- `icon.png` + `icon.svg` auto-handled for browser fallback
- **Redesigned default error page** (500 page)

**Experimental features:**

- `unstable_catchError()` -- granular error boundaries at component level
- `unstable_retry()` in `error.tsx` -- built-in retry with `router.refresh()` + `reset()`
- `experimental.prefetchInlining` -- bundles all segment data into single response
- `experimental.cachedNavigations` -- caches static/dynamic RSC data from navigations
- `experimental.appNewScrollHandler` -- reworked scroll/focus management

**AI-focused:**

- `AGENTS.md` included in `create-next-app` templates
- Browser log forwarding
- `next-browser` (experimental)

**Turbopack:** 200+ bug fixes, SRI support, `postcss.config.ts` support, tree shaking improvements, Server Fast Refresh

### Nx impact (HIGH)

- **Must update generators** for `proxy.ts` instead of `middleware.ts`
- Turbopack as default -- webpack config in executors should use `--webpack` for legacy
- `cacheComponents` replaces PPR experimental flags
- `next lint` removal -- Nx lint target should invoke ESLint directly
- Async request APIs enforcement -- migration generator opportunity
- Node 18 drop aligns with Nx requirements
- **16.2 Adapters API** -- evaluate if `@nx/next` should leverage adapters for custom build output
- **16.2 performance gains** are automatic; no plugin changes needed
- **`transitionTypes`** on `<Link>` is opt-in; no generator changes required
- **`postcss.config.ts`** support in Turbopack -- existing TS PostCSS configs now work

| 7.3.1             | Jan 7   | Last v7 patch      |

### Vite 8.0.0 stable -- KEY EVENT

**Rolldown is now the single bundler**, replacing both esbuild (dep optimization) and Rollup (production builds).

**Breaking changes:**

- **`build.rollupOptions` -> `build.rolldownOptions`** (rollupOptions deprecated but compat layer auto-converts)
- **`worker.rollupOptions` -> `worker.rolldownOptions`**
- **`optimizeDeps.esbuildOptions` deprecated** -> `optimizeDeps.rolldownOptions` (compat layer auto-converts)
- **Lightning CSS** is now a normal dependency (default for CSS minification)
- **`import.meta.hot.accept` no longer accepts URLs** -- must pass id
- Package resolution: when both `browser` and `module` fields exist, uses `resolve.mainFields` order
- Extglobs not yet supported; TypeScript legacy namespace only partially supported
- **Install size ~15 MB larger** (~10 MB lightningcss, ~5 MB rolldown)

**Migration reality:** A compatibility layer auto-converts existing `esbuildOptions` and `rollupOptions` to their Rolldown/Oxc equivalents. Many projects will work without config changes. However, migration is recommended.

**Migration paths:** Direct upgrade OR gradual via `rolldown-vite` package.

### Nx impact (URGENT -- now stable)

- **`@nx/vite` must support Vite 8 as primary** -- this is no longer beta
- Compat layer means existing configs won't break immediately, but generated configs should use `rolldownOptions`
- Any Nx code referencing `esbuildOptions` for dep optimization needs updating
- Lightning CSS default may affect projects with custom CSS processing
- `rolldown-vite` package useful for Nx migration generators
- **v7.3.1 is the last v7** -- plan deprecation timeline
- Test all Vite-dependent plugins: `@nx/vite`, `@nx/remix`, `@nx/vue`, `@nx/nuxt`

### Rspack 2.0 breaking changes (HIGH IMPACT)

**Infrastructure:**

- **Node 18 dropped** -- Node 20+ required
- **Pure ESM packages** -- `@rspack/core` and `@rspack/cli` ESM-only

**Config renames/removals:**

- `output.hotUpdateGlobal`: `webpackHotUpdate*` -> `rspackHotUpdate*`
- `output.chunkLoadingGlobal`: `webpackChunk*` -> `rspackChunk*`
- `experiments.rspackFuture` removed; `bundlerInfo` moves to `output`
- `experiments.lazyCompilation`, `experiments.parallelLoader`, `experiments.outputModule` removed
- `.swcrc` reading disabled by default
- `output.charset` removed
- Legacy `output.library*` options removed
- `WarnCaseSensitiveModulesPlugin` removed
- `plugin.getHooks()` removed

**Default changes:**

- `builtin:swc-loader` enables `verbatimModuleSyntax: true` by default
- `module.parser.javascript.exportsPresence` defaults to `'error'`
- `asyncWebAssembly` enabled by default
- CSS enabled by default
- `target` cascades to SWC and LightningCSS defaults (configure once)

**New in beta.5+:**

- RSC compatible with lazy compilation
- CJS require destructuring tree shaking
- `import.defer()` support
- `cache.portable` configuration

### Nx impact (HIGH)

- **ESM-only is the biggest change** -- all `require()` calls to rspack packages must become dynamic `import()` or the plugin must be ESM
- Global renames (`webpack*` -> `rspack*`) affect HMR/chunk loading and Module Federation
- `.swcrc` disabled by default -- configs relying on it need explicit opt-in
- `verbatimModuleSyntax` default may cause TS import issues
- Node 18 drop aligned with ecosystem direction
- Beta.7 released today (Mar 17) -- stable approaching, likely Q2 2026

## Cross-Cutting: TypeScript 6.0 Beta

**Released:** February 11, 2026

| Change                               | Impact                                               |
| ------------------------------------ | ---------------------------------------------------- |
| `strict` defaults to `true`          | Existing projects may break without explicit `false` |
| `module` defaults to `esnext`        | Generator tsconfig updates needed                    |
| `target` defaults to `es2025`        | Safe for modern projects                             |
| `types` defaults to `[]`             | Projects relying on ambient `@types/*` will break    |
| ES5 target deprecated                | Breaks legacy configurations                         |
| `--outFile` removed                  | Breaks bundled output projects                       |
| `--moduleResolution classic` removed | Must migrate to `node`/`bundler`/`nodenext`          |
| `--baseUrl` deprecated               | Use `paths` with `#/` imports                        |
| `esModuleInterop: false` forbidden   | Must be true or omitted                              |

### Nx action required

- Audit all tsconfig generators across plugins
- Ensure migration generators handle new defaults
- Test all framework plugins with TS 6 beta
