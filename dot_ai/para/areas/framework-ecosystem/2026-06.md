## Frameworks

### Angular

- **Releases this period**: 22.0.0 GA (June 3), 22.0.1, 22.0.2, 22.0.3, **22.0.4** (latest stable, June 26). 22.1.0-next.3 in `next`.
- **22.0.3 / 22.0.4 patches**: Bug fixes only; no API surface change. Node requirement tightened: `^22.22.3 || ^24.15.0 || >=26.0.0` (note: Node 22.22.0 and 22.22.1 no longer listed; exact LTS patch matters).
- **22.1.0-next progression**: next.1 -> next.3 in 11 days. No breaking surface visible yet in search results; likely still early feature accumulation.
- **Key changes**:
  - Signal Forms, `resource`/`httpResource`/`rxResource` async APIs, `@angular/aria` all graduated to stable.
  - Selectorless components: import directly in templates without selector string.
  - OnPush change detection is now the default (migration writes opt-out code via `ng update`).
  - `strictTemplates` and incremental hydration also now on by default.
  - SSRF + path-hijacking guards added to `platform-server`.
  - `ng skills add` (npx skills) ships AI coding guidance for Angular.
  - Requires: TypeScript 6, Node 22.22+ / 24 / 26. Node 20 dropped.
- **@nx/angular impact**: P0 -- open issue #35910 confirms users cannot migrate to Angular 22 today due to peer-dep conflicts in `@nx/angular`. Required work:
  1. peerDep range bump to include Angular 22
  2. Node 22+ / TS 6 gating in generators
  3. FetchBackend default migration
  4. OnPush default migration (ng update handles it in Angular CLI but Nx generator output must match)
  5. Signal-forms generator option
  6. `forms.min/max` string -> number migration
  7. `ComponentFactoryResolver` sweep in scaffolded code
  8. SSRF guards for SSR generator output (platform-server)
- **Action needed**: Yes -- users blocked on migration. Highest priority Nx deliverable this cycle.

### Remix v2 / React Router v7 / React Router v8 / Remix v3

**IMPORTANT: React Router v7 is now superseded by v8 (shipped June 17). The migration story has shifted.**

- **Remix v2 (we use this)**
  - **Latest version**: `@remix-run/react` 2.17.5 (dist-tag `latest`).
  - **Maintenance health**: Active -- 2.17.5 is the most recent stable (last 2 weeks). Nightly and pre tags still publishing. No EOL signals found.
  - **EOL / sunset signals**: None stated. The team continues shipping patches. The open question is whether the team shifts resources to Remix 3.
  - **@nx/remix impact**: Plugin still builds v2 cleanly.

- **React Router v8 (the new migration target -- replaces v7)**
  - **Released**: June 17, 2026. Latest: **8.0.1** (June 18 patch -- removes obsolete AppLoadContext type accidentally left from v7). `dist-tag: latest`.
  - **Breaking vs v7**:
    - ESM-only published package (no CJS export).
    - tsconfig target/lib updated to ES2022.
    - All `future.v8_*` flags from v7 are now defaults (removed from config).
    - CSRF check: bug fix may break apps behind reverse proxies if adapter doesn't set correct host URL. `allowedActionOrigins` config needed.
    - Node support: Active LTS only (Node 22.22.x maintenance, Node 24 active).
  - **Migration story v7 -> v8**: Minimal if all `future.v8_*` flags were adopted. Official blog: "most boring major yet." Codemods available for flag cleanup.
  - **@nx/react-router impact**: ESM-only is the key risk. If Nx Runner or Jest consumers do `require('react-router')` they will break. peerDep range needs to add `^8.0.0`. Generator output for `future.v7_*` flags in RR7 scaffolding must be updated.
  - **Annual release schedule**: Team committed to yearly major releases going forward.

- **Remix v3 (new ground-up framework, NOT a React upgrade)**
  - **Status**: 3.0.0-beta.0 (since April 30). No beta.1 yet in this reporting window.
  - **What it is**: Fully new framework; drops React entirely, uses a Preact fork. Runs on Fetch API (Request/Response), supports Node/Deno/Bun without adapters. Not a drop-in upgrade from v2.
  - **Nx relevance**: Too early for a plugin. Beta is stable enough to track but no stable candidate in sight. Monitor for 3.0.0-beta.1+.

- **Migration recommendation**: Stay on Remix v2 for now. Plan migration path to **React Router v8** (not v7 -- v8 just shipped). `@nx/remix` should surface RRv8 as the recommended upgrade path. Trigger for starting: once `@nx/react-router` supports v8 and the ESM interop issue is resolved.

## Bundlers

### Vite

- **Releases**: 8.0.15 (early June) + 8.0.16 (June 2) + **8.1.0 (June 23)**. LTS: 6.4.3, 7.3.5.
- **8.1.0 key changes**:
  - **Bundled Dev Mode (experimental)**: Previously "bundled-dev" opt-in (8.0.13). Now named and polished. For large apps (10k+ modules): 15x faster dev startup, 10x faster full-page reload, immediate HMR. Enabled via `server.mode: 'bundled'` or similar opt-in.
  - **Lightning CSS integration expanded**: Two new opt-in features that may replace PostCSS in Vite v9. Lightning CSS already handles CSS modules and minification in 8.x; 8.1 deepens the integration (PostCSS still available for custom plugins, e.g. TailwindCSS).
  - **WebAssembly ESM module integration**: Native support for the WASM/ESM proposal (TC39/W3C).
  - **Node requirement updated**: `^20.19.0 || >=22.12.0` (adds Node 20.19.x as acceptable minimum; drops earlier 20.x patches).
  - Rolldown dependency pinned to `~1.1.2` (compatible with 1.1.3 patch).
- **@nx/vite impact**: P1. CNW templates should target 8.1.0. Bundled Dev Mode validation needed against inferred-target snapshot tests -- it changes the dev pipeline significantly. Lightning CSS config options (`css.lightningcss`) may need plumbing through `@nx/vite` config schema. WASM support is additive. Node 20.19+ requirement is narrower than before.

### Rspack

- **Releases**: 2.0.5 -> 2.0.8 (patches) -> **2.1.0** (June 26) -> **2.1.1** (latest).
- **2.1.x key changes**:
  - **`import.meta.glob`**: Now supported natively (matches Vite and Turbopack API). Key migration enabler -- projects using Vite's glob imports can now target Rspack without rewrites.
  - **`pureFunctions` enabled by default in production**: Detects pure function calls via `#__NO_SIDE_EFFECTS__` and `pureFunctions` config. Enabled across module boundaries. May produce different bundle output vs 2.0 if user code has annotated functions previously not tree-shaken.
  - **Rust React Compiler**: SWC bumped to support Rust version of React Compiler. Opt-in via config.
  - **TypeScript 7 support**: Parser and type-strip support for TS7 syntax.
  - **Source Phase Imports**: `import source` syntax supported.
  - **Persistent cache cleanup**: `maxAge` and `maxVersions` (renamed from `maxGenerations`) for automatic stale-cache eviction.
  - **Experimental `runtimeMode`**: New runtime internals for performance.
  - Performance: 20+ optimizations (reduced serde JSON, flag iteration, source map, loader construction).
  - **No breaking changes vs 2.0** (additive features only).
- **@nx/rspack impact**: P0 carryover (v2 integration still open from April). New additions for 2.1: pureFunctions-default-on needs snapshot regression coverage (behavior change in production output). `import.meta.glob` support is a positive enabler for Vite-to-Rspack migration generators. TypeScript 7 compat is additive.

### Rolldown

- **Current**: **1.1.3** (patch from 1.1.2; Vite 8.1.0 pins `~1.1.2` so compatible).
- **1.1.x changes**: `experimental.lazyBarrel` now **defaults to `true`**. Lazy barrel optimization skips compiling barrel-file re-exports that are never used. Meaningful speedup for large component libraries. Output is unchanged for most projects, but barrel-heavy codebases (Ant Design, MUI icons) may see different tree-shaking results.
- **Opt-out**: `lazyBarrel: false` (deprecated in a future release).
- **@nx/vite impact**: P1 -- transitive through Vite. Snapshot regression testing for projects with large barrel imports. Update Rolldown docs/integration notes from "1.0 stable" to "1.1.x with lazyBarrel-on-by-default."
- **Nx relevance**: Still no standalone Rolldown plugin needed yet.

### esbuild

- **Current**: 0.28.1 (minor patch; was 0.28.0).
- **Key changes**: No details found; minor release, no breaking API changes expected.
- **@nx/esbuild impact**: Bump peer range. Carryover: Go 1.26 GC validation + text-import feature surface still pending from May.

### webpack

- **Current**: **5.108.1** (minor version, from 5.107.2).
- **5.108.x key changes**:
  - **ES module auto-detection**: Top-level `await` and `import.meta` now act as ES module markers (no explicit `type: "module"` needed), matching Node.js behavior.
  - **Bun target**: Emits ESM, externalizes `bun:*` and Node.js built-ins.
  - **Deno target**: `deno`, `deno2`, `deno1.40` -- emits ESM, resolves Node built-ins via `node:` specifiers, keeps `npm:`/`jsr:`/`http(s)://` external.
  - **Universal target preset**: Browser + worker + Node.js + Electron + NW.js; always outputs ESM.
  - **`output.html`**: Automatically emits HTML files per entry point with injected JS/CSS chunks (built-in alternative to `HtmlWebpackPlugin`).
  - **`output.workerChunkFilename` + `entry.worker`**: Dedicated worker chunk naming.
  - **CSS enhancements**: `[hash]` in `localIdentName` now resolves to local ident hash; `[modulehash]` is the module hash. `css-url` HTML source type for URL extraction from CSS attributes.
  - **HMR for HTML modules**: Body + title DOM patching on update.
  - **BREAKING**: `MODULE_NOT_FOUND` runtime guard decoupled from `output.pathinfo` -- moved to its own dedicated option. Projects relying on `output.pathinfo` to gate this guard will see behavior change.
- **@nx/webpack impact**: P1. New targets (Bun, Deno, Universal) should be exposed in `@nx/webpack` generators for users building for these runtimes. `output.html` is a generator opportunity (replace HtmlWebpackPlugin usage). Audit whether any Nx executor logic relies on `output.pathinfo` for the `MODULE_NOT_FOUND` guard -- breaking change potential.

### SWC

- **Current**: **1.15.43** (from 1.15.41; two patches).
- **Key changes**: SWC was bumped by Rspack 2.1 to support the Rust React Compiler. No known breaking changes in the 1.15.x line.
- **@nx/js / @nx/next impact**: None for standalone use. If consuming via Rspack 2.1, the React Compiler support is now available.

## Cross-Cutting Themes

### 1. Major-version graduation wave landing
Angular 22 GA + React Router v8 in the same two-week window is the biggest Nx compat event of 2026 so far. Both have known gaps in Nx plugins (peer dep conflict for Angular 22; ESM-only for RR8). This needs coordinated triage.

### 2. AI/MCP integration is now table stakes
Angular 22 ships `provideWebMcpTools` + `ng skills add`. Next.js 16.3 ships built-in MCP for AI coding agents. Two major frameworks now have first-class MCP in the same month. Nx's own MCP story needs a competitive audit.

### 3. ESM-only as the new baseline
React Router v8 ESM-only joins Rspack 2.0 (ESM-only package), Rolldown (ESM), and Vite (ESM). The Nx plugin ecosystem still has CJS consumers that `require()` these. A systematic audit of `@nx/*` plugin import patterns for ESM-only deps is warranted.

### 4. Node 20 end of support convergence
Angular 22 requires Node 22+. Nuxt 4.4.8 requires Node 22.12+. React Router v8 requires Node 22.22+ (maintenance) or Node 24. Vite 8.1 now accepts Node 20.19+ as well as 22.12+. Nx should plan Node 22 as the floor for the next major version -- Node 20 is increasingly unsupportable.

### 5. Vite security hardening
Two consecutive patches (8.0.15, 8.0.16) targeting path-traversal-adjacent attack surfaces. Vite is tightening its file-serve security model. Plugin authors using Vite's dev server should re-audit any custom middleware.

### 6. Bundler runtime target expansion (W26 new)
webpack 5.108 added Bun and Deno targets alongside the new Universal preset. Rspack 2.1 added `import.meta.glob` (Vite parity) and TypeScript 7 support. The bundler tier is rapidly expanding its runtime compatibility surface -- `@nx/webpack` and `@nx/rspack` generators need to keep pace.

### 7. Lightning CSS becoming the standard (W26 new)
Vite 8.1 deepens Lightning CSS integration with two new opt-in features positioned as the PostCSS migration path for v9. This affects any `@nx/vite` tooling that plumbs CSS pipeline configuration. Projects using custom PostCSS plugins (TailwindCSS) are unaffected, but vanilla CSS projects should migrate.
