### Major correction from April scan

The April report missed the **AI agent skill** in v2.2. Re-reading the v2.2 announcement (April 13 blog post: "moon v2.2 - Daemon, AI skills, async graph building, async affected tracking, and more"):

- **`debug-task` AI agent skill** -- shipped in v2.2.0 (April 12). 5-step diagnostic workflow for failing or misbehaving moon tasks. Walks through:
  1. Task configuration review
  2. Verbosity/log inspection
  3. Cache state analysis
  4. Problem matching against 15+ known issue patterns (caching, configuration, execution, CI discrepancies)
  5. Fix validation
- Distributed as an AI skill that AI coding assistants (Claude Code, Cursor, etc.) can invoke when a moon task fails.

This is **the first AI-native feature in any direct Nx competitor** that isn't just "we have an MCP server." It's a packaged diagnostic capability targeting the highest-friction moment in any build tool: "my task isn't working and I don't know why."

### Strategic Implications for Nx (delta vs April)

**Threat level: Medium-High (upgraded from Medium)** -- the AI debug skill changes the picture more than the perf numbers did.

1. **Moon shipped the AI-debug story Nx has been talking about.** Nx has the MCP server and `nx fix-ci` (in nx-cloud client-bundle) but not a packaged "diagnose any failing task" skill. Moon's `debug-task` is conceptually adjacent to what `nx fix-ci` does for CI failures, but moon's is broader (any task, any context) and AI-skill-distributed.
2. **Recommended action**: Treat this as a competitive forcing function for the AI roadmap. Evaluate whether Nx should ship a packaged `debug-task` equivalent -- could be an extension of the existing MCP server's diagnostic surface, or a standalone skill. The competitive narrative "Nx has the deepest workspace intelligence" only holds if Nx also has the deepest *diagnostic* surface for that intelligence.
3. **Moon is quiet on releases** (13 days) but the v2.2 feature set is meaningful -- this isn't slowdown so much as a digest period after a major minor.
4. Other v2.2 highlights still stand from April scan: async graph builder (40-170% in larger workspaces -- exact number now confirmed via blog), async affected tracker (up to 150%), unstable daemon (long-lived process keeping graph hot, async file-watch graph rebuilds).

### NEW since April scan: 8.7.0rc1 release notes populated

The 8.7.0rc1 notes were "still being updated and cleaned up" at last scan. Now confirmed contents:

- **`--rewind_lost_inputs`** (experimental) -- LTS backport from 9.1.0. Rerun actions within a single build to recover from remote/disk cache evictions.
- **`--experimental_remote_cache_chunking`** -- LTS backport from 9.1.0. Read/write large blobs to/from remote cache in chunks.
- **NEW (8.7.0-only, not in 9.1.0)**: **SOCKS proxy support** -- `socks://`, `socks4://`, `socks5://` URLs in `HTTPS_PROXY`/`HTTP_PROXY` env vars and related system properties. Targets enterprise networks behind SOCKS proxies.
- **NEW**: `strip_components` attribute on `extract`/`download_and_extract`/`http_archive` -- strip path components when extracting (familiar to anyone who's used `tar --strip-components`).
- Cross-platform launcher fix (Windows binary on Linux), `package_group` external repo labels, Starlark `splitlines()` U+0085 fix.

### Strategic Implications for Nx

**Threat level: Low for JS/TS (unchanged).** SOCKS proxy + `strip_components` are enterprise-network polish. The chunking and rewinding flags continue the industry-wide attack on large-blob cache transfer (alongside Turbo's 256KB chunk bump). No JS/TS crossover.

## Gradle

**Latest stable**: **v9.5.0 (April 28, 2026)** -- NEW (ends the 40-day silence flagged last month)
**Prior**: v9.4.1 (March 19), v9.4.0 (March 4), v9.3.1 (Jan 29)
**LTS**: 8.14.4 (Jan 23)

### Key changes in Gradle 9.5.0 (April 28) -- Minor

**Diagnostics & Reporting**:

- **Task provenance in error messages** -- failures now reveal whether a task came from a build script, settings script, or plugin. Example: "Execution failed for task ':app:compileJava' (registered by plugin 'org.gradle.api.plugins.JavaPlugin')." This is the kind of error-message investment that pays for itself in user trust.
- **Daemon JVM mismatch diagnostics** -- when a daemon can't be reused due to JVM incompatibility, Gradle now logs concrete wanted-vs-actual JVM configuration mismatches at INFO level (replacing vague "couldn't reuse daemon" messages).

**Build Authoring & Plugin Development**:

- **Type-safe accessors for Kotlin Settings convention plugins** -- IDE autocomplete and compile-time validation in precompiled settings plugins. Catches the "I typo'd a settings property" class of bug at compile time.
- Wrapper download retries -- `gradle-wrapper.properties` now configures automatic retry for distribution downloads.
- **`disallowChanges()`** on Domain Object Collections -- plugin authors can lock collections to prevent unauthorized modifications.
- `gradle init --into <dir>` -- target directory for new project init.

**Network**:

- **`GRADLE_DAEMON_BIND_ADDRESS`** env var -- explicit client-daemon communication address. Targets restrictive enterprise networks.

**IDE & CI**:

- Tooling API exposes help and version info -- IDEs (IntelliJ) can now execute `--help` and `--version` through standard Tooling API actions.
- **`--develocity-url` CLI flag** -- automatically publish Build Scans to a Develocity instance without project modification. **Strategic**: lowers the friction for adopting Develocity to "one CLI flag" instead of "modify your build script". Direct parallel to Turbo's OAuth move and Nx Cloud's `nx connect` flow.

**DX**:

- **Reorganized `--help` output** -- CLI options grouped by category (Built-in, Execution, Configuration, Performance, Security, Diagnostics, Logging, Develocity) instead of alphabetical. Surface area indication: Gradle's CLI is large enough that alphabetical was hurting discoverability.

### Blog activity in window

- **"Choice, Clarity, and the Future of Caching in Gradle Actions"** (April 3) -- continued GHA caching focus (already noted last scan).
- **"GitHub Actions for Gradle v6: What's Changing and Why"** (March 24) -- already noted.
- No new May 2026 posts as of scan date.

### Strategic Implications for Nx (delta vs April)

**Threat level: Low (direct), Medium (feature-parity pressure)** -- Gradle is back in active release cadence after the 40-day silence. Nothing in 9.5.0 is a JS/TS competitive threat, but **two patterns matter**:

1. **`--develocity-url` flag** is a friction-reduction story for Gradle's enterprise intelligence platform that mirrors what Turbo just shipped (OAuth/device flows for Vercel remote cache) and what Nx Cloud has had. Three competitors converging on "minimize friction from `init` to `connected to platform`" inside two months. Nx Cloud's onboarding flow should be benchmarked against the post-`--develocity-url` Gradle flow and the post-OAuth Turbo flow.
2. **Task provenance in errors** is a small but high-trust DX investment. Nx's error messages are generally good but worth auditing -- when a task fails in Nx, do users know whether the task came from an inferred plugin, an explicit `project.json`, a target default, or a generator-produced config? If the answer is "not always," that's a Gradle-comparable improvement worth scoping.

## Cross-Competitor Themes (delta vs April)

1. **Friction-reduction for platform onboarding is now a 3-of-5 pattern.** Turbo (OAuth/device flow shipped in v2.9.7 stable, May 1), Gradle (`--develocity-url` shipped in 9.5.0, April 28), Nx Cloud (already had it). All three competitors converged on the same UX problem inside the same six weeks. **Nx Cloud's onboarding lead is no longer differentiating; it's table stakes.** The next competitive surface is what happens *after* connection (intelligence quality, AI features, fix automation).
2. **AI-native features start appearing in product, not just MCP.** Moon shipped a `debug-task` AI skill in v2.2 (April 12). This is the first packaged AI-diagnostic capability in a direct competitor. Turbo and Bazel still have nothing equivalent. Nx has the MCP server and `nx fix-ci` -- adjacent but not packaged as discrete AI skills for assistants to invoke.
3. **Supply-chain hardening burst.** Turbo's 2.9.9 canaries (May 4) include scoped npm publishing credentials, hardened release workflows, signed OG images. Gradle's "Avoiding the Next Supply Chain Disaster" blog (Feb 25) preceded this. Pants quietly cherry-picked Trivy v0.69.2 into its 2.30 line in April. Industry-wide hygiene push, likely reactive to broader npm incidents (Nx itself was a victim in Aug 2024).
4. **Large-repo perf is still the universal theme** (carryover from April). Turbo `--affected`+`--filter` combinable, 256KB chunk size, prune correctness. Moon async graph builder (40-170%), async affected tracker (up to 150%). Bazel `--rewind_lost_inputs`, remote cache chunking. Nx's "we've always had this" positioning needs refreshed benchmarks to stay credible.
5. **Cadence resumption.** Turbo's 21-day silence resolved (8 stable + canaries in 4 days). Gradle's 40-day silence resolved (9.5.0 ships). Moon and Pants are now the quiet ones (13d and 18d respectively).
