## Vite+ (VoidZero / now Cloudflare)

> **Highest-priority section this cycle -- escalated threat.**

**Previous status:** Alpha, watch lightly.
**Current status:** Beta (July 2, 2026), Cloudflare-backed. Active competitive threat.

### Cloudflare Acquisition (June 4, 2026)

VoidZero announced it was joining Cloudflare on June 4, 2026. All team members joined Cloudflare. Key commitments:

- **Open source maintained:** Vite, Vitest, Rolldown, Oxc, and Vite+ remain open source, vendor-agnostic, and community-driven.
- **$1M Vite ecosystem fund** committed by Cloudflare, administered by the Vite core team.
- Strategic rationale from Cloudflare: "build the future of the AI-native web." Vite integration with Cloudflare Workers/Pages/D1/R2 expected.
- The acquisition removes the startup survival risk. Vite+ now has enterprise-scale backing.

### Vite+ Beta (July 2, 2026)

**CLI:** `vp` (Vite+)
**Components:** Vite 8, Vitest, Rolldown, tsdown, Oxlint, Oxfmt + built-in task runner
**Stats:** 180+ fixes since alpha; 1,300+ public repos depend on vite-plus

Key features relevant to Nx/monorepo competition:

- **`vp run` -- Monorepo-aware task runner with intelligent caching.**
  Powered by `vite-task` (Rust, 453 stars, 548 commits), a standalone monorepo task runner that fingerprints input files to determine cache eligibility -- no explicit `inputs` configuration required. If inputs are unchanged, task is skipped instantly from cache.
- **Dependency-aware scheduling.** Workspace ordering derived from `package.json` dependency relationships, not a separate task graph config. This is a simplification vs. Nx (`project.json` targets) and Turbo (`turbo.json` pipeline).
- **`vp dev`:** Starts a dev server with HMR via Vite 8.
- **`vp check`:** Format (Oxfmt) + lint (Oxlint) + typecheck in one command.
- **Framework-agnostic, MIT licensed.** Works for CLIs, libraries, and web apps.
- No explicit monorepo config required for basic caching -- this is the differentiating DX claim.

**What `vite-task` does NOT yet have:**

- No stable release (pre-release state); the Vite+ beta bundles it but it has no independent versioned artifact yet.
- No distributed task execution / cloud cache (analogous to Nx Cloud or Turbo Remote Cache). Not yet announced.
- No equivalent to Nx generators or Moon templates for scaffolding.

### Nx implications

**Threat level: HIGH (escalated from MEDIUM/WATCH).**

1. **The "will it survive?" concern is resolved.** Cloudflare backing eliminates the startup risk that made Vite+ easy to dismiss. It will keep shipping.
2. **Automatic input fingerprinting is a compelling DX story.** Nx requires explicit `inputs` configuration for fine-grained cache correctness. Vite+ claims to infer them automatically. If this works in practice for complex workspaces, it removes a common Nx friction point. **Recommended: evaluate vite-task's fingerprinting behavior on a mid-size Nx workspace and document where it falls short.**
3. **Vite+ targets the same "start a JS project" audience as `create-nx-workspace`.** The unified `vp` CLI (runtime + package manager + build + test + lint + task runner) is a compelling zero-config entry point. Nx's entry point requires more configuration to reach the same productivity baseline. **This is a positioning threat at the top of the funnel.**
4. **`vite-task` is Rust-based, matching Nx's own native approach.** Performance parity is plausible; benchmarks will matter.
5. **No cloud/remote cache yet.** This is Nx's window. Vite+ users who scale will need remote caching and DTE. Nx Cloud can capture that upgrade path if positioned early.
6. **Cloudflare integration risk.** Vite+ + Cloudflare Workers + Cloudflare R2 (cache storage) is a natural product bundle. If Cloudflare ships a hosted remote cache backed by R2 and integrated with their developer platform, it competes with Nx Cloud in a Cloudflare-centric stack.

## Turborepo

**Latest stable:** v2.10.3 (~July 3, 2026)
**Prior stable at last scan:** v2.9.18 (June 10, 2026)
**New stable minor:** v2.10.0 (June 26, 2026) -- milestone shift from 2.9.x series

### Released this period

**v2.10.0 (June 26, 2026) -- New minor: local cache eviction, deferred input hashing**

- **Local cache eviction:** New `cacheMaxAge` and `cacheMaxSize` turbo.json options. Users can now cap how old cached entries are and how much disk space the local cache consumes. This was a long-standing user request (issue #7029). Nx has no equivalent local cache size controls.
- **Deferred input hashing:** `inputs` entries can now be structured objects that defer file hashing until upstream task dependencies have completed. Tasks depending on a deferred task also defer their hash computation. Reduces wasted hashing for tasks whose inputs are generated by upstream tasks.
- **Composable `--affected` and `--filter`:** These flags now combine, allowing filtered + affected intersection queries (e.g. run affected tasks in a specific scope).
- **TUI improvements:** Toggle between interactive TUI and streamed log output; click rows in the TUI task list to select tasks; auto-copy selection to clipboard on mouse release.
- **Graceful shutdown for watch mode:** Multiple reliability fixes for `turbo watch` -- graceful shutdown, respecting task inputs, restarting deferred hash consumers.

**v2.10.1, v2.10.2, v2.10.3 (June 28 - ~July 3)**

- Transitive dependency fixes in engine graph pruning
- Lockfile preservation during pruning for npm and pnpm
- Recovering from slow initial hash (timeout -> retry instead of hard fail)
- Additional TUI stability
- Windows environment variable fixes

**v2.10.1-canary series (June 25+)**

Active canary development continues; no single breakthrough feature.

### Key changes by category

- **Features:** Local cache eviction (cacheMaxAge/cacheMaxSize), deferred input hashing, composable flags
- **Performance:** Deferred hashing reduces wasted work; hash timeout recovery prevents CI stalls
- **DX:** TUI click-to-select, log toggle, graceful watch shutdown
- **AI:** None. `experimentalCI` object remains canary-only and undocumented (15+ weeks).
- **Security:** No new security releases since v2.9.18 (June 10)

### Nx implications

**Threat level: HIGH (sustained).**

1. **3.0 is dead or rebranded.** The March 25 "preparing for 3.0" blog is now 15 weeks stale. Turbo shipped v2.10 instead. No 3.0 RC. The version milestone no longer represents a strategic pivot; it's incremental.
2. **Local cache eviction (cacheMaxAge/cacheMaxSize)** is a concrete Nx gap. Users with large workspaces and limited disk accumulate stale local cache entries. Turbo now has configurable eviction; Nx does not. This will appear in "Turbo vs Nx" comparison discussions. **Recommended: add local cache size/age controls to Nx backlog.**
3. **Deferred input hashing** is an architectural improvement that could reduce CI slowness on complex task graphs. Nx's input hashing model should be evaluated against this approach.
4. **TUI improvements** continue to widen the gap vs. Nx's terminal experience. Nx Console is the IDE story; the terminal-only experience is less polished.
5. **`experimentalCI` remains undocumented at 15 weeks.** If it surfaces as a "Turbo Cloud" or "managed CI" product, it competes with Nx Cloud DTE.

## Cross-Competitor Themes (July 2026)

1. **Vite+ exits the "watch passively" tier.** Cloudflare backing + beta = this is now a real product with real funding. The Vite ecosystem (Vite, Vitest, Rolldown, Oxc) now has a monorepo task runner story. Three funded JS/TS monorepo tools compete for the same audience: Nx, Turborepo, Vite+.

2. **Automatic vs. explicit task configuration is the new DX battleground.** Vite+ claims zero-config input fingerprinting. Turborepo introduced deferred input hashing to reduce explicit config burden. Nx requires the most explicit configuration of the three. This framing will appear in evaluation posts and comparisons. Nx needs either better inference or a stronger counter-narrative (e.g., "explicit is correct; auto-inference makes wrong cache decisions").

3. **Local cache management is now a Turbo differentiator.** `cacheMaxAge`/`cacheMaxSize` fills a gap that Nx hasn't addressed. Small but visible in large-workspace discussions.

4. **Cloudflare + Vite+ = potential remote cache threat.** Cloudflare R2 is a natural backend for a hosted task cache. If Cloudflare ships "Vite+ Remote Cache," backed by R2 and integrated with Cloudflare Workers pipelines, it enters Nx Cloud's territory. No announcement yet, but the product surface is obvious.

5. **Moon v2.3 series stabilizing; v2.4 is the next watch point.** CAS cache experiment has been live for 5 weeks through v2.3.5 without reversion. v2.4 promotion to stable is the expected outcome.

6. **Turbo 3.0 appears abandoned in favor of 2.x continuations.** This removes the "big 3.0 launch" pressure from Turbo's narrative. Turbo is now a slow, reliable incrementalist. The urgency premium on Nx responding to "Turbo 3.0" is gone.

### Where Nx Is Behind

- **Local cache eviction:** Turborepo added `cacheMaxAge`/`cacheMaxSize` in v2.10.0. Nx has no equivalent. Small but visible.
- **Zero-config input inference:** Vite+ claims automatic fingerprinting with no `inputs` config needed. This is the most significant new gap -- it attacks a known Nx friction point.
- **MCP generator discoverability:** Moon's `get_template`/`get_templates` tools target the AI code-generation workflow. Nx generators are not yet MCP-queryable with schema introspection.
- **Cache security audit:** Two competitors patched cache path traversal and manifest manipulation on June 10. Nx has not publicly confirmed auditing the same classes (carryover from June).

### Where Nx Is Ahead

- **Distributed Task Execution:** No competitor ships DTE. Turbo's `experimentalCI` is 15 weeks undocumented.
- **Nx Cloud:** Remote caching with security controls, observability, and managed CI orchestration. Vite+ has no cloud product. Moon has no cloud product.
- **Nx Console:** Turbo contracted scope (removed web UI). Vite+ has no IDE extension. Nx Console is the richest IDE-integrated monorepo experience.
- **Generator ecosystem breadth:** Nx generators (create-nx-workspace, @nx/* plugins) vastly exceed Moon's templates and Vite+'s current scaffold story.

### Unassailable Moats

- **DTE + Nx Cloud:** Architectural decision requiring distributed coordination infrastructure. No competitor can replicate with a single release.
- **Workspace generator ecosystem:** Decade of community plugins and official generators.

## Recommended Actions (July 2026)

**High urgency:**

1. **Evaluate Vite+ Beta's automatic input fingerprinting.** Run `vite-task` on a mid-size Nx workspace and document where auto-inference is wrong, incomplete, or breaks cache correctness. The goal: either identify gaps that make Nx's explicit `inputs` model superior, or identify what Nx can borrow. This is the most concrete new competitive capability this cycle.
2. **Assess Vite+ as a top-of-funnel threat at `create-nx-workspace`.** Vite+ targets the same "start a JS project" audience with a simpler entry point. Review whether `create-nx-workspace` can be streamlined for Vite-centric projects, or whether a "Nx + Vite+" integration story is worth pursuing.
3. **Audit Nx cache implementation for path traversal and manifest manipulation (carryover from June, now 4 weeks outstanding).** Two competitors patched cache security vulnerabilities on June 10. This is now overdue.

**Medium urgency:**

4. **Add local cache size/age eviction controls to Nx backlog.** Turborepo shipped `cacheMaxAge`/`cacheMaxSize` in v2.10.0. This will appear in comparison posts. A simple `cache.maxAge`/`cache.maxSize` option in `nx.json` would close the gap.
5. **Watch for Vite+ remote cache / Cloudflare R2 integration.** The Cloudflare acquisition makes a hosted remote cache product a natural next step. Monitor for announcements. If it ships, it enters Nx Cloud territory for Cloudflare-centric shops.
6. **Track Moon v2.4.** Expected late July. Watch for CAS cache promotion to stable (from experiment). If promoted, Moon will market "local/remote cache format unification" as a differentiator.
7. **Assess MCP generator discoverability gap (carryover).** Moon's `get_template`/`get_templates` tools target the AI code-generation workflow. Nx generators are not yet MCP-queryable with schema.

**Low urgency / monitoring:**

8. **Turbo 3.0 de-prioritize.** 15+ weeks stale; v2.10 milestone suggests 3.0 is not coming. Remove from watch queue. Turbo is now an incrementalist.
9. **Gradle 9.6.1 + config cache marketing:** Wave complete; no new moves expected until Gradle 10 preview.
10. **Pants v2.33 stable:** Expected early-mid July. No JS/TS crossover; light tracking.
11. **Evaluate Cloudflare/Vite+ Nx plugin opportunity.** Given Cloudflare's stated commitment to open source and vendor-agnosticism, a first-party `@nx/vite-plus` or `@nx/cloudflare` integration could capture users who adopt Vite+ but want Nx's generator/DTE story.
