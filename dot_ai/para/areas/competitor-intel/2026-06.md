### NEW since June-8 scan

**v2.9.17 (June 9) -- JIT task input hashing**

- **`$TURBO_JIT$` pre-release feature:** marks file inputs to be hashed just-in-time, after the task's `dependsOn` tasks have completed. Solves the generated-file correctness problem: without JIT, generated files are hashed at startup using stale/missing content, causing incorrect cache hits. With `$TURBO_JIT$`, the final cache key for a task is deferred until just before it runs. Tasks using `$TURBO_JIT$` show a deferred hash message in `--dry` output. Feature marked pre-release; surface may change before stable.
- Improved pnpm dependency handling
- Peer dependency filtering improvements
- Test infrastructure optimizations (multiple)

**v2.9.18 (June 10) -- Artifact integrity + auth security**

- **Artifact signature field separation:** splits artifact signature fields to prevent key confusion in cache entries
- **OidHash hex buffer validation:** validates content-addressable ID buffers before use, preventing cache corruption from malformed entries
- **Block self-hosted login URLs from attempting Vercel's SSO:** prevents a misconfiguration class where self-hosted Turbo Remote Cache instances attempt Vercel's SSO flow
- Cache restoration improvements (consolidation from preceding canaries)
- Security hardening focus -- no major new features

**v2.9.19-canary.1 through canary.5 (June 11-14) -- Windows PTY + pnpm**

- **Auth HTTP timeouts** (canary.1, re-landed after v2.9.15 batch)
- **Affected root task detection in query mode** (canary.1)
- **Windows graceful shutdown** (canary.1, re-landed)
- **ConPTY Ctrl+C signal delivery to child processes during graceful shutdown** (canary.3) -- Windows PTY hardening
- **PTY support for interactive Windows tasks** (canary.4) -- enables interactive terminal input on Windows
- **Caching outputs through internal symlinks** (canary.5) -- correctness for symlinked output paths
- **Filtered pruned pnpm workspace patches** (canary.5) -- pnpm correctness
- **ConPTY cursor query output prevention on Windows** (canary.5)
- PR workflow timeouts added to CI (canary.3)
- Shim updated: no longer terminates local turbo instances on Ctrl+C (canary.2)

### Key changes by category

- **DX:** Windows PTY improvements -- interactive task support, ConPTY signal handling. Meaningful improvement for Windows mono-repo users.

### NEW since June-8 scan

**v2.3.3 (June 10) -- Remote cache security fix**

- **SECURITY:** Fixed an issue where task outputs being hydrated from the remote cache can overwrite files outside the declared output list, if the manifest in the remote cache has been compromised. This is a path-traversal / zip-slip class vulnerability: a malicious or compromised cache server could craft a manifest pointing output paths outside the project, allowing arbitrary file writes on the developer's machine or CI host.
- **Go toolchain:** Fixed a regression in `go.mod` parsing that failed on `tool ()` declarations (the parenthesized tool directive syntax introduced in Go 1.24).

### Key changes by category

- **Security:** Path-traversal vulnerability in remote cache hydration -- highest severity fix in the Moon v2.3 series.
- **Ecosystem:** Go `go.mod` tool() directive parsing regression fixed.

### Nx implications

**Threat level: HIGH (unchanged, pattern now established -- Moon ships features fast and patches security quickly).**

1. **The Moon v2.3.3 security fix should prompt an Nx cache extraction audit.** The vulnerability class (path traversal during cache artifact hydration) is not unique to Moon's CAS implementation -- any tool that unpacks a cache archive without strict output-path validation is exposed. **Recommended: audit Nx's local cache restore, Nx Cloud cache download, and `--remote-cache` code paths for path-traversal safety.** Specifically: verify that no path in a restored cache entry can escape its declared output directory (strip leading `/`, block `..` traversal, validate against declared outputs list before write).
2. **The CAS cache experiment (v2.3.0) introduced this bug and it was caught in 9 days.** This is actually evidence that Moon's security response is fast (same-day or next-day fix after report). But it also signals that new architectural features (CAS, unified local/remote cache format) expand attack surface.
3. **MCP template tools (v2.3.0) still active.** No v2.4 announcement yet; v2.3 patch track suggests v2.4 is ~2-4 weeks out. The MCP generator discoverability gap (Moon ahead) from the June-8 report stands.

## Cross-Competitor Themes (delta vs June-8 scan)

1. **Remote cache security is now a confirmed competitive surface.** Moon's v2.3.3 patches a path-traversal in cache hydration. Turbo's v2.9.18 patches artifact integrity (OidHash, signature separation). Two competing tools patching cache-layer security in the same week is a signal: build cache security is under scrutiny (likely from security researchers targeting the ecosystem post-Turbo's March CVEs). Nx should be proactive here, not reactive.
2. **JIT input hashing (Turbo) and CAS local cache (Moon) are two different bets on the same problem** -- correctness of the task input hash when generated files are involved. Turbo defers hashing of specific inputs. Moon unifies local/remote cache formats (CAS). Both are attacking the "stale cache key from generated outputs" problem. Nx's approach (project graph dependency tracking) may already solve this -- worth documenting.
3. **Windows PTY depth (Turbo canary) is accelerating.** Four consecutive canaries (v2.9.19-canary.1-5) addressing Windows PTY edge cases (interactive tasks, Ctrl+C signal, ConPTY query artifacts) suggests a major Windows improvement shipping in v2.9.19 stable. For JS-monorepo users on Windows, this could be a meaningful DX improvement that positions Turbo better against Nx on Windows.
4. **Gradle 9.6.0 final is the nearest milestone** (1-2 weeks out). Expect the config cache marketing push to intensify. Prepare benchmark response now.
