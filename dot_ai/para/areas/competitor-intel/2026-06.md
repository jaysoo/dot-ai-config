**v2.10.0 (June 24, 2026) -- MINOR RELEASE: Local cache eviction, JIT hashing, --affected+--filter**

The "3.0" narrative (first hinted in March 2026, ~16 weeks ago) was quietly abandoned; Vercel shipped a minor version bump instead:

- **`cacheMaxAge` for local cache age eviction:** Entries older than the configured duration (e.g., `"30d"`, `"2w"`) are removed at the start of each `turbo run`. Human-readable duration strings (s/m/h/d/w). Also configurable via `TURBO_CACHE_MAX_AGE` env var.
- **`cacheMaxSize` for local cache size eviction:** When the local cache exceeds the configured size (e.g., `"10GB"`, `"1.5TB"`), oldest entries are evicted first. Configurable via `TURBO_CACHE_MAX_SIZE` env var. If both are set, age eviction runs before size eviction.
- **JIT task input hashing:** Tasks can defer input-hash computation until after their dependencies complete. `--dry=json` reports `hash: null` and a `hashReason` for JIT tasks since the final hash is unknown until execution time. Enables scenarios where task outputs feed into downstream input hashes.
- **`--affected` + `--filter` can now be combined:** Previously mutually exclusive. `turbo run build --affected --filter='./packages/*'` now works as expected.
- **pnpm v11 multi-document lockfile handling:** Improved parsing for pnpm v11's new lockfile format.
- **Bun workspace lockfile pruning + nested dependency preservation** (carried from canary stream).
- **npm workspace-nested package support** during pruning.
- **@turbo/codemod handles package manager catalogs in migration** and supports `turbo.jsonc` files.
- **18 first-time contributors** in this release -- community momentum signal.
- Security/hardening items from the canary stream (daemon IPC endpoint hardening, cache filesystem boundary enforcement, OTel endpoint validation, auth callback state validation, symlink defense during cache restore) are included.
- Remote cache upload chunk size increased from 8KB to 256KB.

### Nx implications (updated June 25)

**Threat level: HIGH (elevated from prior sub-period).**

1. **"3.0" narrative confirmed dead -- shipped 2.10.0 instead.** 16 weeks after "preparing for 3.0," Vercel quietly dropped the major version bump. The window for Nx counter-positioning on "Turbo 3.0 delays" is now closed; the narrative has resolved as a minor release.
2. **`cacheMaxAge`/`cacheMaxSize` is a concrete feature gap for Nx.** Turbo now has built-in local cache eviction with both age-based and size-based policies, configurable via env vars. Nx's local cache does not have equivalent eviction controls. Developers hitting disk-full CI issues or stale cache problems will notice. **Recommended: assess adding local cache eviction to Nx (or the Nx daemon) as a DX improvement.**
3. **`--affected` + `--filter` combination** closes a known Turbo usability gap. Nx has had equivalent functionality for longer (`nx affected` with `--projects` filter). Validate that Nx's combination behavior is at least as capable.
4. **JIT task input hashing** is a model difference worth understanding. Turbo is now deferring hash computation for tasks whose inputs depend on prior task outputs. Nx's current model hashes task inputs at scheduling time. JIT hashing enables scenarios like "task B's input hash includes task A's outputs" -- potentially more correct incremental behavior. **Recommended: determine if Nx's task input model supports the same patterns or if there's a correctness gap.**
5. **TUI migration to Ghostty** (canary) signals Turbo is investing in modern terminal output. Ghostty's escape code handling is more performant than vt100. Nx's task runner TUI (via @nx/tui) should track compatibility.
6. **Web UI removal** is a scope contraction (June 10). Nx Console remains the richer IDE-integrated experience. Net positive for Nx positioning.
7. **Two security batches in 4 weeks (v2.9.14 + v2.9.18)** = Turbo is under active external security research. Nx should maintain equivalent incident response posture.
8. **Git argument injection fix (#13055)** is a class worth auditing in Nx's own git operations (e.g., `nx affected --base <branch>`, remote-cache tag metadata).
9. **`experimentalCI` object config** still ships in canaries without docs. When it surfaces, assess against DTE/Nx Cloud CI story.
10. **Rsbuild examples** confirm Turbo is targeting Rspack/Rsbuild users. Nx has `@nx/rspack` -- check parity holds.

**v2.3.0 (June 1) -- MAJOR RELEASE: CAS cache, native hashing, MCP tools, task tags**

This is the most significant Moon release since v2.2.0 (April 12). Key features:

- **CAS Output Cache Experiment:** Task outputs stored in local content-addressable storage, matching the remote cache format. Enables deduplication across tasks and a unified local/remote cache structure. Enable via `experiments.casOutputsCache`.
- **Native File Hashing Experiment:** Custom native hashing implementation running in the task pool, targeting 10-50% performance improvement. Enable via `experiments.nativeFileHashing`.
- **MCP Template Tools:** Two new tools added to Moon's MCP server: `get_template` (inspect a template and its variable schema) and `get_templates` (list all available templates). Enables AI assistants to discover code generation templates before generating code.
- **Task Tags:** Tasks now support a `tags` setting and `#tag` syntax for target addressing (e.g., `app:#quality`). `--tags` option added to `moon query tasks`. `options.mergeTags` controls inheritance.
- **Cache Strategy for Task Dependencies:** New `cacheStrategy` field controls invalidation behavior per dependency: `hash` (default for output-declaring deps), `ignored` (default for others), or `outputs`. Reduces unnecessary cache invalidations for lint/test tasks.
- **Task Target Memory:** 50-100% reduction in memory footprint per task target.
- **SHA256 Git Support:** Prepares for Git's eventual SHA1 -> SHA256 migration.
- **Deno v2.8 Support:** `deno ci` for installs, `deno.lock` in CI, `--prod` flag, `catalog:` reference resolution.
- **Python UV Pip:** New unstable `uv-pip` package manager option.
- **Daemon Improvements:** Auto-clean and webhooks now run from daemon instead of at process end.
- **Glob regression fix:** Unbounded walks up to 10x slower -- fixed.

**v2.3.1 (June 4) -- Daemon + toolchain**

- In-memory caching for toolchain operations (locating executables) -- faster tool resolution
- Daemon startup faster: workspace graph loaded in background after server readiness
- Plugin distribution moved to ghcr.io (away from raw URLs)
- Daemon connect/ready logic fix for potential Windows connection issues
- `cacheStrategy` inference fix when experiments enabled
- Leftover locks at `.moon/cache/locks` now cleaned up

**v2.3.2 (June 5) -- Daemon hardening + Go**

- Pipeline continues if daemon client cannot connect (no hard failure)
- Go: `inferRelationshipsPackages` setting for `go list --deps` package patterns

- `project:^` input resolution bug fixed for overlapping project sources

**v2.3.3 (June 10) -- SECURITY: Remote cache path traversal**

- **Security fix: task outputs being hydrated from the remote cache can overwrite files outside the output list if the manifest in the remote cache has been compromised.** This is a path traversal-adjacent vulnerability -- an attacker with write access to the remote cache can manipulate the manifest to cause arbitrary file writes on the developer or CI machine running moon. No CVE assigned in release notes. Severity: moderate to high depending on remote cache trust model.
- Go toolchain regression: `go.mod` parsing of `tool ()` blocks (introduced in v2.3.0-v2.3.2) corrected.

### Key changes by category

- **Features:** Task tags, cacheStrategy, SHA256 git, UV pip unstable (v2.3.0)
- **Performance:** CAS cache (experiment), native file hashing 10-50% (experiment), 50-100% task memory reduction, glob fix (up to 10x), in-memory toolchain caching (v2.3.0-2.3.1)
- **DX:** Deno v2.8, UV pip, daemon UX, Python `--no-dev` default, Rust `--locked` default (v2.3.0-2.3.4)
- **Ecosystem:** ghcr.io plugin distribution, Go v1.24 + inferRelationshipsPackages (v2.3.0-2.3.2)
- **AI:** `get_template` + `get_templates` MCP tools -- AI agents can now discover and inspect Moon's template library (v2.3.0)
- **Security:** Remote cache manifest path traversal patched (v2.3.3) -- no CVE; no advisory number

### Nx implications

**Threat level: HIGH (unchanged).**

Moon jumped from six weeks of bug-fix-only to a major minor in one cycle. Several features have direct Nx relevance:

1. **CAS local cache** is strategically significant. Moon's local cache now matches its remote cache format -- enabling deduplication and a unified caching story. Nx's local cache is not CAS-based. If the Moon experiment stabilizes, Moon will be able to claim "same caching semantics locally and remotely." This is a differentiator claim Nx cannot currently make. **Recommended: investigate whether Nx's roadmap includes CAS-format local cache.**
2. **Native file hashing 10-50% improvement** is a benchmarkable claim. Nx uses Rust via the Nx daemon for file hashing too. If Moon can ship a concrete benchmark (e.g., "hashes a 50k-file repo 40% faster"), it becomes a sales point in large-workspace conversations. **Recommended: have a comparable Nx file-hashing benchmark ready.**
3. **MCP template tools** directly compete with Nx generators as an AI-discoverable workflow. Moon's AI agents can now list templates, inspect their variable schemas, and generate code via the MCP server. Nx's generators are not yet MCP-discoverable in the same structured way. **Recommended: assess whether Nx Console's or Nx MCP's generator tooling exposes a comparable `list_generators`/`describe_generator` interface.**
4. **Task tags** give Moon a grouping primitive Nx doesn't have as first-class syntax. Nx uses project tags for filtering; Moon's `#tag` syntax targets tasks specifically. Low immediate threat but worth tracking as a DX differentiator.
5. **Daemon resilience (pipeline continues on disconnect)** -- Moon is hardening its daemon story while last scan noted Turbo was removing daemon. Two competing bets: Moon going harder on daemon, Turbo going daemon-free. Nx's daemon-based approach aligns with Moon's direction, not Turbo's.

## Cross-Competitor Themes (updated June 22)

1. **Daemon as infrastructure is bifurcating.** Moon is investing heavily in daemon hardening -- background graph loading, panic recovery, pipeline-continues-on-disconnect, webhooks from daemon. Turbo moved away from daemon. Nx's daemon-based model (Nx Daemon for project graph, local cache) aligns with Moon's bet. This could become a narrative battleground: Moon/Nx claim daemon gives richer runtime context; Turbo claims simplicity.
2. **CAS (content-addressable storage) enters the local cache space.** Moon's experiment brings CAS semantics to local task outputs. This mirrors how remote caches (Nx Cloud, Turbo Remote Cache) work. If CAS-local becomes the standard, tools without it will need to explain why. First mover in the local CAS space is Moon.
3. **MCP is becoming a build-tool battleground.** Moon now has: run_task, sync_projects, get_touched_files, get_template, get_templates. That's 5 MCP tools for AI agent integration. The addition of template-discovery tools specifically targets the AI code-generation workflow. This is the clearest indication that Moon sees AI-native workflows as a competitive axis, not just a feature.
4. **AI code generation discoverability** (templates/generators) is the new front. Moon's MCP can now answer "what can I generate?" and "what variables does this template take?" before the AI generates code. This reduces hallucination risk in AI-driven scaffolding. Nx generators are richer but not yet MCP-discoverable in this structured way.
5. **Cache security is now a cross-competitor concern.** Both Turborepo (v2.9.18, June 10) and Moon (v2.3.3, June 10) patched cache-related security vulnerabilities on the same day. Turbo fixed 7 issues including git argument injection and cache archive path escape. Moon fixed a remote cache manifest poisoning enabling arbitrary file writes. This validates that remote and local caching surfaces are under active security research. Nx's cache implementation should be audited for the same classes of bugs: manifest path traversal, symlink escape, special mode bit preservation.
6. **Enterprise crypto (P-521 ECDSA) in Turbo canary; self-hosted SSO isolation in v2.9.18.** Two enterprise signals in the same release cycle. Turbo is reaching into enterprise certificate requirements AND isolating self-hosted deployments from Vercel's SSO. Suggests Turbo is actively selling to enterprise self-hosted accounts.
7. **Gradle 9.6.0 config cache marketing completed.** RC1 to final in 23 days (May 28 - June 20). The CI-specific property-tracking fix is now stable and marketable. Gradle's config-cache marketing wave (4 touchpoints since April: blog, 9.5 headline, RC1, 9.6 final) is now complete. JVM-shop prospects have been primed. Nx needs comparable numbers.
8. **Rust is the default for performance-critical internals.** Pants moving PathGlobs to Rust (June 12) joins Moon's native file hashing (experiment), Turbo's Rust core, and Nx's native bindings. The entire build-tool ecosystem is converging on Rust for I/O-bound primitives.

### Where Nx Is Behind

- **Local cache eviction:** Turbo v2.10.0 (June 24) ships `cacheMaxAge`/`cacheMaxSize` with env-var overrides. Nx has no equivalent. Developers with full-disk CI or stale cache issues will notice. This is now a concrete, marketable DX gap.
- **Local CAS cache:** Moon's CAS experiment (v2.3.0, June 1) is stabilizing -- v2.3.3/2.3.4 hardened it without reverting it. If v2.4 promotes it to stable, Nx will need a response or a counter-narrative.
- **MCP depth for AI agents:** Moon's MCP server covers task running, project sync, affected detection, and template discovery (5 tools total). Nx generators are not yet MCP-queryable with schema introspection. Gap is widening.
- **Cache security audit:** Two competitors patched cache path traversal and manifest manipulation on June 10. Nx has not publicly confirmed auditing the same classes. This should be prioritized.
- **JIT task input hashing model:** Turbo v2.10.0 introduces deferred hash computation for tasks whose inputs depend on prior task outputs. Whether Nx's task scheduling model already handles this pattern correctly is worth verifying.

### Where Nx Is Ahead

- **Distributed Task Execution:** No competitor ships DTE. Turbo's `experimentalCI` is now 14 weeks in canary with no stable.
- **Nx Cloud:** Managed CI orchestration with cache, security controls, and observability. Turbo has remote cache; Moon has no cloud product.
- **Nx Console maturity:** Turbo contracted scope by removing web UI mode (June 10). Nx Console remains the richer IDE-integrated experience.
- **Generator ecosystem:** Nx generators (create-nx-workspace, @nx/* plugin generators) are far more extensive than Moon's templates. The gap is discovery, not breadth.

### Unassailable Moats

- **DTE + Nx Cloud:** Architectural decision requiring distributed coordination infrastructure. Moon cannot replicate with a daemon-level change; Turbo's `experimentalCI` is still undocumented at week 14.
- **Workspace generator ecosystem breadth:** Decade of community plugins and official generators. Moon templates are much fewer.
