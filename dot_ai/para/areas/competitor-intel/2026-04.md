## Executive Summary

- **Turborepo shipped the 2.9 marketing release (March 25)** with "up to 96% faster Time to First Task," `turbo query` GraphQL stable, incremental-adoption support via cycle tolerance, and a sweeping deprecation list staging for 3.0. Followed by a rapid v2.9.1 -> v2.9.6 stable cadence through April 10 and v2.9.7 canaries today. This is the most aggressive marketing push from Turbo since 2.0. ([blog](https://turborepo.dev/blog/2-9), [v2.9.0](https://github.com/vercel/turborepo/releases/tag/v2.9.0))
- **Moon jumped to v2.2.0 (April 12)** introducing a long-running background **daemon** (`moon daemon`) for file watching and cache invalidation, an **async affected tracker** (100-150% faster), and an **async project/task graph builder** (100-170% faster on large workspaces). This is the clearest "catch up to Nx daemon" move we've seen from Moon. ([v2.2.0](https://github.com/moonrepo/moon/releases/tag/v2.2.0))
- **Bazel 9.0.2 (April 9)** and **9.1.0rc1 (April 9)** -- 9.1 introduces `--experimental_remote_cache_chunking` and `--rewind_lost_inputs` (rerun actions to recover from cache eviction mid-build). Enterprise-grade resilience features. ([9.1.0rc1](https://github.com/bazelbuild/bazel/releases/tag/9.1.0rc1))
- **Threat level: HIGH -> HIGH (unchanged, reasserted).** Turborepo's 2.9 launch closed several gaps we were relying on (cache eviction/TTL via `cacheMaxAge`/`cacheMaxSize`, `--affected` + `--filter` composition, circular-dep tolerance), plus planted a 3.0 flag that will dominate narrative the rest of 2026. Moon's daemon lands them in the "daemon + async graph" camp that was previously Nx-distinct.
- **Cross-competitor theme: "agent-first" and "CI as first-class surface."** Turborepo added `experimentalCI` task config, `@vercel/agent-readability`, and cursor[bot] contributions; Pants added an AI tooling notice for contributions; 3 of 5 tools tightened CI/agent-facing surfaces since Feb 12.

Earlier v2.8.x stables were covered in [2026-03.md](./2026-03.md). This report focuses on **2.9.x**.

### Key changes in 2.9.x

#### Features (v2.9.0 marketing highlights)

- **Time to First Task: 80-96% faster** across tested repos (Vercel backend 91%, Vercel frontend 81%, create-turbo 80%). Source: [blog](https://turborepo.dev/blog/2-9). This is startup overhead, not cache/execution speed -- but it's the number they'll market.
- **`turbo query` promoted to stable** with two first-class subcommands: `turbo query affected` (structured JSON for CI) and `turbo query ls`. GraphQL against the Package and Task Graph. They are explicitly positioning this as agent-friendly: "coding agents are incredibly adept at writing queries."
- **Circular package-dependency tolerance** (v2.9.0). Turborepo now validates the Task Graph instead of the Package Graph, allowing package-level cycles. Pitched as an adoption aid for monorepos that can't fix cycles up front.
- **`cacheMaxAge` and `cacheMaxSize`** for local cache eviction (v2.9.1, [PR #12487](https://github.com/vercel/turborepo/pull/12487)). Closes a real gap.
- **Incremental task caching** (v2.9.4, [PR #12531](https://github.com/vercel/turborepo/pull/12531)).
- **Circular package dependency detection in boundaries** (v2.9.5, [PR #12567](https://github.com/vercel/turborepo/pull/12567)).
- **`--affected` and `--filter` can be combined** (v2.9.4, [PR #12543](https://github.com/vercel/turborepo/pull/12543)). Closes a composition gap that users hit frequently.
- **Two-dot git ranges in filter selectors** (v2.9.7-canary.1, [PR #12599](https://github.com/vercel/turborepo/pull/12599)).

#### Performance

- **Remote cache upload chunk size raised 8KB -> 256KB** (v2.9.5, [PR #12568](https://github.com/vercel/turborepo/pull/12568)).
- **`boundaries` checking parallelized with Rayon, DFS traversals cached** (v2.9.5, [PR #12569](https://github.com/vercel/turborepo/pull/12569)).

#### DX

- **Actionable error when remote cache requested but not linked** (v2.9.6, [PR #12584](https://github.com/vercel/turborepo/pull/12584)).
- **Concurrency error references `turbo.json`** directly (v2.9.6, [PR #12582](https://github.com/vercel/turborepo/pull/12582)).
- **Custom CA cert loading** in fast webpki HTTP client (v2.9.6, [PR #12591](https://github.com/vercel/turborepo/pull/12591)) -- enterprise proxy/TLS support.
- **Watch loop unblocked for interruptible persistent tasks** (v2.9.2, [PR #12509](https://github.com/vercel/turborepo/pull/12509)).

#### AI / Agent

- **`@vercel/agent-readability` replaces local ai-agent-detection** (v2.9.4, [PR #12528](https://github.com/vercel/turborepo/pull/12528)). Turbo is centralizing agent-aware content rewriting into a Vercel-owned library -- suggests Vercel is building a reusable agent-readability primitive across their docs/SDK surface.
- **`cursor[bot]` making direct commits** to turborepo examples (v2.9.7-canary.1). Vercel is using Cursor agents to maintain the repo. Signal, not feature.
- **Turborepo Agent Skills** referenced in the 2.9 blog -- coordinated with `turbo query` stable to pitch "agents querying your monorepo."

#### Deprecations (prep for 3.0)

This is the most strategic part of 2.9. Per the [release blog](https://turborepo.dev/blog/2-9), deprecated in 2.9 (to be removed in 3.0):

| Deprecated | Replacement |
|---|---|
| `turbo-ignore` | `turbo query affected` |
| `turbo scan` | (removed, functionality obsolete) |
| `--parallel` | `persistent` + `with` in turbo.json |
| `--no-cache` | `--cache=local:r,remote:r` |
| `TURBO_REMOTE_ONLY`, `--remote-only` | `--cache=remote:rw` |
| `TURBO_REMOTE_CACHE_READ_ONLY`, `--remote-cache-read-only` | `--cache=local:rw,remote:r` |
| `TURBO_DAEMON`, `--daemon`, `--no-daemon`, `daemon` in turbo.json | **Daemon no longer used** |
| `--graph` with png/jpg/pdf | svg/html/mermaid/dot |
| `--graph` with json | `turbo query` |
| `--scope` for `turbo prune` | positional args |

Two takeaways:

1. **Turborepo is removing its daemon entirely** in 3.0. Their startup-overhead story (96% faster TTFT) is now supposed to be good enough without one. This is the opposite of Moon's move (which just added a daemon) and re-emphasizes Nx's daemon as a differentiated bet. Worth testing whether Turbo's startup actually matches Nx daemon-mode on warm invocations.
2. **`--cache` as unified cache-mode flag** consolidates 5+ env vars/flags into one composable string. Good DX move; Nx should evaluate whether our cache controls (`--skip-nx-cache`, `NX_NO_CLOUD`, etc.) would benefit from similar consolidation.

### Nx implications

- **`--affected` + `--filter` composition gap closed** -- users asking for this on our side now have a Turbo answer. We should confirm our own behavior matches or exceeds.
- **`turbo query` stable** is now a marketable agent surface. Our MCP tools (`nx-mcp`) cover the same ground but we should ensure parity messaging: we are already shipping MCP-native, not GraphQL. That's arguably a better agent surface (agents don't need to know GraphQL). Consider a positioning post: "Why Nx chose MCP over GraphQL for agent queries."
- **Daemon removal in 3.0** reframes Nx's daemon as either (a) a strategic advantage (better warm-path performance) or (b) legacy weight if Turbo's TTFT claims hold. We should benchmark Nx cold/warm vs Turbo 2.9 cold in realistic monorepos before Turbo 3.0 lands.
- **Circular package dependency tolerance** removes a pain point in adoption from Turbo-land. Worth checking whether `nx import` and `nx init` on cyclic workspaces behaves gracefully.
- **`cacheMaxAge` / `cacheMaxSize`** -- do we have declarative equivalents? If not, consider adding to `nx.json` caching config.

## Moon

**Latest stable**: v2.2.0 (2026-04-12) -- jump from v2.1.4
**Releases this period**: v2.0.0 (Feb 18), v2.0.1, v2.0.3, v2.0.4, v2.1.0 (Mar 16), v2.1.1, v2.1.2, v2.1.3, v2.1.4 (Apr 4), **v2.2.0** (Apr 12). 10 stable releases in 60 days.

### Key changes

#### Features

- **Daemon (unstable)** in v2.2.0. `moon daemon start/stop`, file watcher, cache invalidation. Setting: `unstable_daemon` in `.moon/workspace.*`. [v2.2.0 notes](https://github.com/moonrepo/moon/releases/tag/v2.2.0). **Headline feature and a direct structural catch-up to Nx.**
- **Async affected tracker** (100-150% faster) gated behind `experiments.asyncAffectedTracking` (v2.2.0).
- **Async project/task graph builder** (100-170% faster on large workspaces) gated behind `experiments.asyncGraphBuilding` (v2.2.0). Does not support cycles in async mode.
- **On-demand affected marking** (v2.1.2) -- tracker marks projects/tasks as affected lazily to short-circuit early.
- **`affectedFiles` settings**: `filter`, `ignoreProjectBoundary`, `passDotWhenNoResults` (v2.1.0).
- **`runInSyncPhase`** task option (v2.1.0) -- run tasks during `moon sync`.
- **Environment detection now recognizes remote devboxes** (Codespaces, Gitpod) -- v2.1.0.
- **TypeScript v6 support** (v2.1.2).
- **`pruneProjectReferences`** setting (v2.1.0).
- **System toolchain now built-in** (not WASM) -- v2.2.0. Boot-time improvement.

#### Performance

- Graph memory footprint reduction (nodes as integers, v2.2.0).
- Plugin registry faster loading (v2.2.0).
- Task output archiving off main thread pool (v2.2.0).

#### Breaking

- **`--json` output format changed** for `moon action-graph`, `moon project-graph`, `moon task-graph` in v2.2.0. Nodes are now integers + separate `data` map. Consumers must update.

### Nx implications

- **Moon finally has a daemon.** This converges Moon's architecture with Nx's core differentiator. On our side, we should emphasize daemon maturity: our daemon is stable, theirs is unstable-flagged. But the gap is closing.
- **Moon's async graph building is directly competitive with Nx's Rust-based project graph.** We're both in the "background thread pool per project" camp. We should verify our project graph build times stay ahead in the 500+ project range.
- **Minor opportunity**: Moon's removal of WASM for the system toolchain is an admission that WASM-plugin overhead hurts. Nx uses native modules and is well-positioned here.
- **Moon is investing heavily in polyglot toolchain inference** (Go `go list --deps`, Python extras metadata, Rust Cargo, TS v6) -- same space as Nx's `@nx/gradle`, `@nx/dotnet`, etc.

## Bazel

**Previous stable in period**: 8.6.0 (Feb 26), 9.0.1 (Mar 10), 9.0.2 (Apr 9)

### Key changes

#### 9.0.2 (patch) - [release notes](https://github.com/bazelbuild/bazel/releases/tag/9.0.2)

- Local and remote repo contents cache now include host OS + CPU architecture in cache key ([#29194](https://github.com/bazelbuild/bazel/pull/29194)). Correctness fix.
- Include verification aware of Windows case-insensitivity ([#29214](https://github.com/bazelbuild/bazel/pull/29214)).
- `RUNFILES_TREE_MARKER` treated as directory in `BuildEvent.forArtifact()` ([#29010](https://github.com/bazelbuild/bazel/pull/29010)).
- Platform-specific startup bazelrc flags ([#29251](https://github.com/bazelbuild/bazel/pull/29251)).

#### 9.1.0rc1 (new minor) - [release notes](https://github.com/bazelbuild/bazel/releases/tag/9.1.0rc1)

- **`--experimental_remote_cache_chunking`** -- read/write large blobs to remote cache in chunks ([#28903](https://github.com/bazelbuild/bazel/pull/28903)). Requires server support. Large-artifact remote cache performance.
- **`--rewind_lost_inputs` (experimental)** -- rerun actions within a single build to recover from cache eviction ([#28958](https://github.com/bazelbuild/bazel/pull/28958)). Enterprise resilience feature. If you're on a hot-eviction cache, Bazel can now recover mid-build.
- **Persistent worker protocol forwarding** (`persistentWorkerProtocol` platform property) for intermixing JSON/Proto workers across a build ([#28710](https://github.com/bazelbuild/bazel/pull/28710)).
- `compatibility_level` and `max_compatibility_level` attributes in MODULE.bazel are now no-ops ([#28616](https://github.com/bazelbuild/bazel/pull/28616)).
- Remote repo contents cache supports all reproducible repo rules ([#29225](https://github.com/bazelbuild/bazel/pull/29225)).

### Nx implications

- **`--rewind_lost_inputs` is interesting**. Nx Cloud cache eviction doesn't currently auto-retry mid-run. Worth a product conversation about whether we surface a similar capability.
- **Remote cache chunking** -- relevant to Nx Cloud's remote cache artifact sizing. Track whether customers with large artifacts (docker layers, large bundle outputs) are hitting transfer limits.
- Bazel remains the enterprise-resilience benchmark. Nothing here that threatens Nx's JS/TS/polyglot positioning.

## Cross-Competitor Themes

### 1. Agent-aware product surfaces (4/5 competitors)

- Turborepo: `@vercel/agent-readability` library, cursor[bot] committing, Agent Skills with `turbo query`.
- Moon: MCP `generate` tool (bug fix in v2.1.0 implies Moon already has MCP surface).
- Pants: AI tooling notice for contributions (governance, not product yet).
- Bazel: no explicit agent move.
- Gradle: no explicit agent move.

**Implication**: Agent-first product design is now table stakes. Nx's MCP-first position (nx-mcp) is well-aligned, but we need parity content on "why MCP beats GraphQL for agents" now that Turbo is marketing GraphQL + AI agents together.

### 2. CI-as-first-class configuration (Turbo + Moon)

- Turbo: `experimentalCI` key in turbo.json (v2.8.22 canary -> shipped in 2.9.0).
- Moon: `MOON_PIPELINE_*` env var overrides (v2.2.0).

Both adding CI-specific task/behavior differentiation. Nx Cloud already differentiates CI via DTE/NxAgents -- but the config-level CI surface is a new convention we should evaluate.

### 3. Daemon convergence (Moon arrives, Turbo leaves)

- Moon: adds unstable daemon in 2.2.0.
- Turbo: deprecating daemon entirely in 3.0.
- Nx: daemon is stable and core.

Genuine split in the industry. Whether daemons win depends on whether Turbo's "cold-start fast enough" claim survives real-world testing. Worth a benchmark post.

### 4. Async graph building (industry converging)

- Moon: `asyncGraphBuilding` experiment (v2.2.0).
- Turbo: existing Rust-native graph.
- Nx: Rust-native project graph shipped.

Converging industry bet. Nx is on the right side of this.

### 5. Cache eviction controls

- Turbo: `cacheMaxAge` / `cacheMaxSize` (v2.9.1).
- Bazel: `--rewind_lost_inputs` recovers from eviction (9.1.0rc1).
- Nx: `nx reset` + manual controls.

Nx should evaluate declarative eviction policy in `nx.json`.

|---|---|---|---|
| Daemon | Moon v2.2.0 unstable, Turbo removing in 3.0 | Ahead (stable, mature) | Low -- defend with benchmarks |
| Project graph (Rust) | Moon async v2.2.0 (experimental), Turbo has Rust | Contested | Low |
| Affected analysis | Turbo `turbo query affected` stable + `--affected`/`--filter` compose | Contested | Medium -- verify parity |
| Cache eviction policy (declarative) | Turbo `cacheMaxAge`/`cacheMaxSize` (v2.9.1) | Behind | Medium |

| Startup performance (TTFT) | Turbo claims 80-96% improvement | Contested -- needs benchmark | **High** |

### Where Nx Is Behind (new this month)

- **Declarative cache eviction policy**: Turbo v2.9.1 added `cacheMaxAge`/`cacheMaxSize`. Nx relies on manual `nx reset`. Low effort to close.
- **`--affected` + `--filter` composition**: Turbo shipped in v2.9.4. Verify Nx parity; if missing, add.
- **Cold-start TTFT claims**: Turbo's 80-96% number is a marketing asset. Nx needs a counter-benchmark in warm (daemon) and cold modes. **Most urgent action.**
- **CI-specific task config key**: Turbo `experimentalCI` is a new surface convention.

### Where Nx Is Ahead

- **MCP-native agent surface**: Turbo is pitching GraphQL; Nx ships MCP. MCP is the emerging standard. Defensible but needs louder positioning.
- **Daemon maturity**: Moon unstable, Turbo removing. Nx stable.
- **Nx Cloud remote execution + DTE**: no competitor ships comparable agent distribution. Bazel remote execution exists but is infra-heavy.
- **Polyglot plugin breadth (JS/TS-first + Gradle/Rust/Python via typed plugins)**: broader than Moon or Turbo.

### Unassailable Moats

- **Nx Cloud DTE / NxAgents** -- remote task distribution with SaaS ops. Not viable for Turbo (no Vercel equivalent), Moon (no cloud), or OSS Bazel BuildBuddy peers (different model).
- **Plugin ecosystem** -- `@nx/*` family integrations with Angular, Next.js, Remix, Vite, Jest, Playwright, etc. Turbo has examples-based patterns, not typed plugins.
- **Nx Agents self-healing / MCP self-healing flow** -- not matched by any competitor.

## Recommended Actions

1. **Benchmark Nx cold + warm TTFT vs Turborepo 2.9.x on realistic monorepos (100, 500, 1000 projects)** and publish. Turbo's "80-96% faster" is marketing; we need a defensible counter-number before Turbo 3.0 lands. **Highest-urgency competitive response this month.**
2. **Write "Why Nx chose MCP over GraphQL for agent queries"** positioning post to counter Turbo 2.9 agent narrative. Lean on the self-healing / MCP standard / no-GraphQL-required angles.
3. **Add declarative cache eviction policy to `nx.json`** (e.g. `cache.maxAge`, `cache.maxSizeBytes`). Small effort, closes a Turbo feature gap.
4. **Verify `--affected` + `--filter` composition in Nx CLI matches or exceeds Turbo v2.9.4.** If gaps exist, close them.
5. **Audit `nx init` / `nx import` behavior on workspaces with cyclic package deps.** Turbo now tolerates these; Nx should not regress relative.
6. **Benchmark Nx daemon stability claims vs Moon v2.2.0 unstable daemon** in a blog post. Moon's arrival validates Nx's bet; capitalize on the timing.
7. **Read the April 3 Gradle Actions caching blog** and assess whether it affects `@nx/gradle` + Nx Cloud positioning.
8. **Track Turbo 3.0 daemon-removal impact** through Q3. If real users file performance issues post-daemon-removal, that's content gold for Nx daemon advocacy.
