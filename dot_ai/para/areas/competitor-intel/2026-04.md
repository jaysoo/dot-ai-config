### Key changes in 2.9.x

#### Features (v2.9.0 marketing highlights)

- **Time to First Task: 80-96% faster** across tested repos (Vercel backend 91%, Vercel frontend 81%, create-turbo 80%). Source: [blog](https://turborepo.dev/blog/2-9). Startup overhead, not cache/execution speed.
- **`turbo query` promoted to stable** with two first-class subcommands: `turbo query affected` (structured JSON for CI) and `turbo query ls`. GraphQL against the Package and Task Graph. Positioned as agent-friendly: "coding agents are incredibly adept at writing queries."
- **Circular package-dependency tolerance** (v2.9.0) -- Turborepo now validates the Task Graph instead of the Package Graph, allowing package-level cycles.
- **`cacheMaxAge` and `cacheMaxSize`** for local cache eviction (v2.9.1, [PR #12487](https://github.com/vercel/turborepo/pull/12487)).
- **Incremental task caching** (v2.9.4, [PR #12531](https://github.com/vercel/turborepo/pull/12531)).
- **Circular package dependency detection in boundaries** (v2.9.5, [PR #12567](https://github.com/vercel/turborepo/pull/12567)).
- **`--affected` and `--filter` can be combined** (v2.9.4, [PR #12543](https://github.com/vercel/turborepo/pull/12543)).
- **Two-dot git ranges in filter selectors** (v2.9.7-canary.1, [PR #12599](https://github.com/vercel/turborepo/pull/12599)).
- **Vercel auth moved to standard OAuth/device flows** (v2.9.7-canary.11, [PR #12526](https://github.com/vercel/turborepo/pull/12526)), legacy Vercel auth compat (canary.12), token recovery across flows (canary.13).
- **pnpm v11 multi-document lockfile support** (v2.9.7-canary.7, [PR #12616](https://github.com/vercel/turborepo/pull/12616)).
- **PTY graceful-shutdown fixes** across canary.8-10.

#### Deprecations (prep for 3.0)

Per the [release blog](https://turborepo.dev/blog/2-9), deprecated in 2.9 (to be removed in 3.0):

| Deprecated | Replacement |
|---|---|
| `turbo-ignore` | `turbo query affected` |
| `turbo scan` | (removed, functionality obsolete) |
| `--parallel` | `persistent` + `with` in turbo.json |
| `--no-cache` | `--cache=local:r,remote:r` |
| `TURBO_REMOTE_ONLY`, `--remote-only` | `--cache=remote:rw` |
| `TURBO_REMOTE_CACHE_READ_ONLY`, `--remote-cache-read-only` | `--cache=local:rw,remote:r` |
| `TURBO_DAEMON`, `--daemon`, `--no-daemon`, `daemon` in turbo.json | **Daemon no longer used** |
| `--graph` with png/jpg/pdf | svg/html/mermaid/dot |
| `--graph` with json | `turbo query` |
| `--scope` for `turbo prune` | positional args |

Two takeaways:

1. **Turborepo is removing its daemon entirely** in 3.0. Their startup-overhead story (96% faster TTFT) is now supposed to be good enough without one. Opposite of Moon's move (which just added a daemon) and re-emphasizes Nx's daemon as a differentiated bet.
2. **`--cache` as unified cache-mode flag** consolidates 5+ env vars/flags into one composable string. Nx should evaluate whether our cache controls (`--skip-nx-cache`, `NX_NO_CLOUD`, etc.) would benefit from similar consolidation.

### Nx implications

- **`--affected` + `--filter` composition gap closed** on Turbo side. Confirm Nx parity messaging.
- **`turbo query` stable** is now a marketable agent surface. Our MCP tools cover the same ground but we should ensure parity messaging: we are already shipping MCP-native, not GraphQL. Consider "Why Nx chose MCP over GraphQL for agent queries" post.
- **Daemon removal in 3.0** reframes Nx's daemon as either (a) a strategic advantage or (b) legacy weight if Turbo's TTFT claims hold. Benchmark Nx cold/warm vs Turbo 2.9 cold in realistic monorepos before Turbo 3.0 lands.

- **v2.9.7 stable cut imminent.** 6-day canary freeze is the strongest cadence signal yet. Pre-position any Nx competitive content; expect Vercel to coordinate a "turbo login + OAuth" platform moment with the release.

### Key changes (v2.2.0 -- headline release)

#### Features

- **Daemon (unstable)** in v2.2.0. `moon daemon start/stop`, file watcher, cache invalidation. Setting: `unstable_daemon` in `.moon/workspace.*`. Auto-starts on first use; CLI falls back to in-process graph if unavailable. - **AI Skills for agents** -- v2.2 ships a `debug-task` skill targeting Claude and compatible AI agents. 5-step diagnostic workflow: inspect resolved task config after inheritance merging, run with max verbosity, inspect cache state and hash manifests, match symptoms to root causes via built-in decision tree, validate the fix. Skills install via CLI and invoke by describing task problems to the AI. **Moon's first explicit agent-facing product surface** and a direct counterpart to Nx's MCP self-healing flow. [Blog](https://moonrepo.dev/blog/moon-v2.2).
- **Async affected tracker** -- 146% speedup at 15k tasks. Gated behind `experiments.asyncAffectedTracking` (v2.2.0). Additional +5-10% in v2.2.1.
- **Async project/task graph builder** -- 170% speedup at 5k projects / 20k tasks. Gated behind `experiments.asyncGraphBuilding` (v2.2.0). Does not support cycles in async mode. Does not cache results yet.
- **On-demand affected marking** (v2.1.2) -- tracker marks projects/tasks as affected lazily to short-circuit early.
- **`affectedFiles` settings**: `filter`, `ignoreProjectBoundary`, `passDotWhenNoResults` (v2.1.0).
- **`runInSyncPhase`** task option (v2.1.0) -- run tasks during `moon sync`.
- **Environment detection now recognizes remote devboxes** (Codespaces, Gitpod) -- v2.1.0.
- **TypeScript v6 support** (v2.1.2).
- **`pruneProjectReferences`** setting (v2.1.0).
- **System toolchain now built-in** (not WASM) -- v2.2.0.

#### Follow-up patches (v2.2.1, v2.2.2, v2.2.3)

- v2.2.1: async affected tracker +5-10%; pnpm v10 multi-document lockfile parsing fix; remote cache errors now treated as cache misses; request retries via proto (3 retries exponential backoff); TypeScript project references prefixed with `./` (TS v7 prep).
- v2.2.2: plugin loading perf regression fix (v2.2.0 graph changes slowed plugin loading); VS Code extension v2-schema update; Python toolchain pip/uv dependency direction fix; `unpack` extension missing-plugin-locator fix.
- **v2.2.3 (NEW): action pipeline deadlock fix under sync-heavy load.**

#### Breaking

- **`--json` output format changed** for `moon action-graph`, `moon project-graph`, `moon task-graph` in v2.2.0. Nodes are now integers + separate `data` map.

### Bazel 9.1.0 stable (NEW this scan)

Promoted from 9.1.0rc3 on Apr 20 after a 2-week RC cycle. No net-new features beyond what was in rc1-rc3 (all covered in prior scans). Headline recap:

- **`--experimental_remote_cache_chunking`** ([#28903](https://github.com/bazelbuild/bazel/pull/28903)) -- read/write large blobs to remote cache in chunks. Requires server support. Large-artifact remote cache performance.
- **`--rewind_lost_inputs` (experimental)** ([#28958](https://github.com/bazelbuild/bazel/pull/28958)) -- rerun actions within a single build to recover from cache eviction. Enterprise resilience feature. **Mid-build cache-eviction recovery is a capability Nx Cloud does not have.**
- **`--test_summary=short_uncached` / `detailed_uncached`** ([#29263](https://github.com/bazelbuild/bazel/pull/29263)) -- filter cached results out of test summaries.
- **Persistent worker protocol forwarding** ([#28710](https://github.com/bazelbuild/bazel/pull/28710)) -- `persistentWorkerProtocol` platform property; intermix JSON/Proto workers across a build.
- **Remote repo contents cache keys include host OS + CPU architecture** ([#29195](https://github.com/bazelbuild/bazel/pull/29195)). Correctness.
- **Remote repo contents cache supports all reproducible repo rules** ([#29225](https://github.com/bazelbuild/bazel/pull/29225)).
- **`package_group` supports external repo labels** ([#28893](https://github.com/bazelbuild/bazel/pull/28893)).
- **`rctx.symlink` implicit watch on copy-fallback** ([#28741](https://github.com/bazelbuild/bazel/pull/28741)).
- **`compatibility_level` / `max_compatibility_level` attributes in MODULE.bazel are now no-ops** ([#28616](https://github.com/bazelbuild/bazel/pull/28616)).
- **CcInfo removal now surfaces errors correctly** ([#28360](https://github.com/bazelbuild/bazel/pull/28360)) -- the symbol was intentionally removed in 9.0.0 but failed silently; 9.0.1 onward surfaces the error. **Users must upgrade `rules_go` >= 0.59.0 and `rules_nodejs` >= 6.7.3.** Significant migration friction for Bazel-on-JS users. **Continues to validate Nx's decision not to compete in the Bazel-plus-rules_nodejs space.**
- **Incompatible Starlark change**: `string.splitlines()` no longer treats u+0085 (NEL) as newline ([#28931](https://github.com/bazelbuild/bazel/pull/28931)).
- **Cross-build launcher fix** ([#29239](https://github.com/bazelbuild/bazel/pull/29239)) -- build Windows binaries on Linux.

### Nx implications

- **9.1.0 going stable is a minor signal** -- no new capability in the 9.1.0 tag vs 9.1.0rc3. The larger fact is **Bazel shipped a minor in 60 days after 9.0.0 launched January**, meaning the 9.x cadence is more active than 8.x was.

- Bazel remains the enterprise-resilience benchmark. Nothing here that threatens Nx's JS/TS/polyglot positioning.

### Gradle 9.5.0-RC3 headline features

Verified directly from the [9.5.0-RC3 release notes](https://docs.gradle.org/9.5.0-rc-3/release-notes.html):

- **Task provenance in reports and failure messages.** Task failures now surface which script (build script / settings script / plugin) and which source registered the failing task. **Debuggability feature that encroaches on Nx's Project Details / cache debugging framing** -- users get "why is this task failing and where did it come from" natively in the build output.
- **Precompiled Kotlin Settings plugins** -- type-safe accessors auto-generated for Settings convention plugins. IDE autocompletion + compile-time validation at the settings level.
- **Wrapper download retries** -- `gradle-wrapper.properties` accepts retry attempts and backoff config. Network resilience for CI.
- **`--develocity-url` CLI flag** -- publish Build Scan without modifying project config. **Signal**: Develocity is being lowered from project-config citizen to CLI-override citizen. Makes Develocity adoption frictionless for agents/CI scripts.
- **`disallowChanges()` on domain object collections** -- plugin authors can lock collections post-configuration.
- **No AI/agent features in 9.5.** Confirmed from RC3 release notes. Gradle continues to concentrate AI surface area in **Develocity** rather than core.

### Key changes (non-core)

**April 3 blog post "Choice, Clarity, and the Future of Caching in Gradle Actions"** by Daz DeBoer ([link](https://blog.gradle.org/choice-clarity-future-caching-gradle-actions)) -- Gradle Actions v6 had introduced proprietary "Enhanced Caching" as default, drawing community pushback about data ownership and OSS requirements. **v6.1.0 response**: (1) "Safe Harbor Clause" -- `"Your code and artifacts belong to you. Period."` explicit metadata/usage-metric-only claims; (2) restored fully open-source `basic` caching via `cache-provider: basic` -- MIT-licensed wrapper around GitHub's native `actions/cache`. Maintainers publicly acknowledged they "missed the mark" on communication. **Both caching modes free for public repos; Enhanced Caching free preview for private repos with future limits for large commercial users.**

### Nx implications

- **Gradle 9.5 RC train stalled at RC3 for 5 days.** If a 9.5.0 stable lands in the next 1-2 weeks, `@nx/gradle` must validate against it. Task provenance in reports may surface via our Gradle plugin's output parsing -- verify no regressions in how we read failure messages.
- **Task provenance is a lightweight version of what Nx Cloud does** (we show full task input diffs / cache miss analysis). Gradle's version is native to the build output; Nx Cloud's is richer but requires the cloud surface. Worth articulating as "native debuggability vs platform debuggability" in Gradle comparative content.
- **`--develocity-url` CLI lowering friction for Develocity adoption** -- Gradle streamlining their paid-tier onboarding. Nx Cloud equivalent already exists (`nx-cloud connect`, `NX_CLOUD_ACCESS_TOKEN`); consider whether our CLI-override story is equivalently clean.
- Supply-chain post aligns with Nx's own NXC-4197 expand-deps supply chain hardening work -- validates the theme.
- **Gradle Actions caching fiasco is instructive for Nx Cloud positioning.** (1) ownership language matters -- "your code belongs to you" should be explicit in Nx Cloud marketing; (2) a free OSS-compatible caching alternative beside the paid product reduces friction; (3) community trust can be broken by default-enabled proprietary tooling. Nx Cloud's current positioning (optional, opt-in) is already aligned.

## Cross-Competitor Themes

### 1. Agent-aware product surfaces (unchanged: 2 of 3 direct competitors shipping)

- Turborepo: `@vercel/agent-readability` library, cursor[bot] committing, Agent Skills with `turbo query`.
- Moon: **AI Skills** system shipped in v2.2 -- `debug-task` skill for Claude and compatible AI agents.
- Pants: AI tooling notice for contributions (governance, not product yet).
- Bazel 9.1.0 stable: no explicit agent move.
- Gradle 9.5.0-RC3: no explicit agent move.

**Implication (unchanged)**: Agent-first product design is **the competitive baseline for monorepo tools as of April 2026**. Nx's MCP-first position (nx-mcp + self-healing CI) remains strongest, but the "we're the only one with agent integration" messaging window has closed. **Accelerate MCP skill breadth and publish comparative content ("why MCP beats GraphQL / why Nx Agents beat debug-skills") within 30 days.**

### 2. CI-as-first-class configuration (Turbo + Moon)

- Turbo: `experimentalCI` key in turbo.json (shipped in 2.9.0).
- Moon: `MOON_PIPELINE_*` env var overrides (v2.2.0).

Both adding CI-specific task/behavior differentiation. Nx Cloud already differentiates CI via DTE/NxAgents -- but the config-level CI surface is a new convention we should evaluate.

### 3. Daemon convergence (Moon arrives unstable, Turbo leaves)

- Moon: adds unstable daemon in 2.2.0. **v2.2.3 deadlock patch in sync-heavy pipeline (NEW)** reinforces "unstable" label.
- Turbo: deprecating daemon entirely in 3.0.
- Nx: daemon is stable and core.

Nx is the only vendor with a stable daemon. Moon's latest deadlock patch is direct evidence for that narrative.

### 4. Async graph building (industry converging)

- Moon: `asyncGraphBuilding` experiment (v2.2.0). Plugin-loading regression (v2.2.2) and action-pipeline deadlock (v2.2.3) both trace back to the v2.2.0 rework.
- Turbo: existing Rust-native graph.
- Nx: Rust-native project graph shipped.

Converging bet. Nx's execution is more mature than Moon's.

### 5. Cache eviction controls

- Turbo: `cacheMaxAge` / `cacheMaxSize` (v2.9.1).
- Bazel: `--rewind_lost_inputs` recovers from eviction (9.1.0 stable).
- Nx: `nx reset` + manual controls.

Nx should evaluate declarative eviction policy in `nx.json`.

### Where Nx Is Behind

- **Declarative cache eviction policy**: Turbo v2.9.1 added `cacheMaxAge`/`cacheMaxSize`. Low effort to close.
- **`--affected` + `--filter` composition**: Turbo shipped in v2.9.4. Verify Nx parity; if missing, add.
- **Cold-start TTFT claims**: Turbo's 80-96% number is a marketing asset. Nx needs a counter-benchmark in warm (daemon) and cold modes. **Most urgent action.**
- **CI-specific task config key**: Turbo `experimentalCI` is a new surface convention.

### Where Nx Is Ahead

- **MCP-native agent surface**: Turbo is pitching GraphQL; Nx ships MCP.
- **Daemon maturity**: Moon unstable + 3 patches in 9 days (incl. v2.2.3 deadlock), Turbo removing. Nx stable. **New evidence this scan reinforces the gap.**
- **Nx Cloud remote execution + DTE**: no competitor ships comparable agent distribution.
- **Polyglot plugin breadth (JS/TS-first + Gradle/Rust/Python via typed plugins)**: broader than Moon or Turbo.

### Unassailable Moats

- **Nx Cloud DTE / NxAgents** -- remote task distribution with SaaS ops. Not viable for Turbo (no Vercel equivalent), Moon (no cloud), or OSS Bazel BuildBuddy peers (different model).
- **Plugin ecosystem** -- `@nx/*` family integrations with Angular, Next.js, Remix, Vite, Jest, Playwright, etc. Turbo has examples-based patterns, not typed plugins.
- **Nx Agents self-healing / MCP self-healing flow** -- not matched by any competitor.
