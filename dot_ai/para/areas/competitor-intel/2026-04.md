### Turbo 2.9 Blog (March 25) -- Strategic Themes

1. **"Up to 96% faster"** -- headline perf claim (specifics behind the blog paywall-ish redirect loop, but corroborated by the v2.8.x perf sprint + Rayon parallelization in v2.9.5 + 256KB chunk size + in-process gix-index migrations from v2.8.x)
2. **`turbo query` graduated to stable** -- now the canonical workspace-intelligence API. `turbo-ignore` deprecated in favor of `turbo query affected`. Consolidation story is complete.
3. **"Easier adoption in large repos"** -- marketing framing. Combined with `cacheMaxAge`/`cacheMaxSize`, circular-dep detection, task-level `--affected`/`--filter` correctness fixes, this positions 2.9 as a "we fixed the large-monorepo sharp edges" release. Directly targets Nx's core market.
4. **"Deprecations to prepare for 3.0"** -- first public signal of Turbo 3.0 planning. `--parallel`, `--scope` (prune), `turbo-ignore`, `turbo scan`, graphviz formats, bespoke AI prompts all already deprecated. 3.0 likely consolidates the query engine as the sole API surface and removes remaining legacy commands.

### Strategic Implications for Nx

**Threat level: HIGH (upgraded from Medium-High).**

Turbo 2.9 is the first release where the "we're catching up to Nx" framing flips -- specific capabilities are now **converged or contested** rather than "emerging":

1. **`--affected` + `--filter` combinable (v2.9.4)** closes one of the last ergonomic gaps vs Nx. Nx users who wanted "affected within this scope" had a cleaner story; Turbo users now do too.
2. **Incremental task caching (v2.9.4)** is a generic capability label; without the PR body it's unclear if this is output-level incremental or something more sophisticated. Worth investigating -- could be partial-output caching, which Nx doesn't have.
3. **`cacheMaxAge`/`cacheMaxSize` (v2.9.1)** adds cache lifecycle primitives at the library level. Nx has per-workspace cache size management but not `@turbo/repository`-style library API.
4. **Circular package-dep detection in boundaries (v2.9.5)** -- Nx has this via `enforce-module-boundaries`, but Turbo now ships a built-in rule. Narrows the boundary/conformance differentiation.
5. **Remote cache upload 8KB→256KB (v2.9.5)** is a 32x chunk size bump. Nx Cloud's upload pipeline should be checked against this; if Nx Cloud is still on small chunks, this is a measurable perf gap on large cache uploads.
6. **Rayon-parallelized boundaries checking (v2.9.5)** signals Turbo applying its Rust perf toolkit to correctness features, not just caching. Boundaries used to be "add-on"; now it's first-class and fast.
7. **Graceful shutdown (v2.9.7-canary.4)** is foundational for long-running CI/persistent tasks. Nx's Rust TUI has this; Turbo closing the gap.
8. **Vercel OAuth/device flows (canary.11)** replaces legacy token auth. For Vercel users this is lower-friction auth to Turbo's remote cache -- **directly competitive with Nx Cloud's auth story**. Vercel gets the same "one click, you're connected" story that Nx Cloud has had.
9. **pnpm v11 lockfile support (canary.7)** -- Turbo adding forward-compat for a pre-release pnpm. Nx should verify equivalent support.
10. **3.0 signaled.** First public mention of Turbo 3.0 planning (blog, March 25). Nx should assume Turbo 3.0 lands within 2026. Expect it to consolidate around the query engine, ship the structured-logging pipeline (`--json`/`--log-file` from v2.8.18), and possibly graduate some of the `futureFlags` (globalConfiguration, longerSignatureKey, filterUsingTasks) into defaults.

**Release velocity remains aggressive**: 9 stable releases in the window (v2.8.21 through v2.9.6), plus 13 canaries of v2.9.7. The v2.8.21→v2.9.0 jump in 2 days shows Vercel is willing to cut minors aggressively.

**Nx should consider**:

- **Check Nx Cloud upload chunk size** against Turbo's new 256KB -- measurable perf benchmark opportunity.
- **Investigate what "incremental task caching" actually means in v2.9.4.** If it's partial-output caching, that's a feature gap; if it's the cache-restore manifest work extended, less urgent.
- **Prep counter-positioning for Turbo 3.0.** Likely lands in 2026. "Nx has had a unified intelligence API for years" + "Nx 22/23 shipped these same capabilities without needing a major" angles.
- **Watch the auth flow change.** OAuth/device-flow parity matters for remote cache adoption; Nx Cloud's signup friction should be benchmarked against Vercel's.

## Moon

**Latest stable**: v2.2.3 (April 21, 2026)
**Timeline**: v2.1.3 (Mar 26) → v2.1.4 (Apr 4) → **v2.2.0 (Apr 12 -- minor)** → v2.2.1 (Apr 16) → v2.2.2 (Apr 20) → v2.2.3 (Apr 21)

### Key Changes in v2.2.0 (April 12) -- Minor

**Breaking**:

- `--json` output for `moon action-graph`, `moon project-graph`, `moon task-graph` changed: graph nodes are now **integers** (not objects), node data in separate `data` object. Memory optimization with breaking-change consequence.

**Major Additions**:

- **Experimental async affected tracker (100-150% faster)** -- `experiments.asyncAffectedTracking` setting. Directly competitive with Turbo's affected perf sprint.
- **Experimental async project/task graph builder (100-170% faster in large workspaces)** -- `experiments.asyncGraphBuilding`. Thread-pool-per-project. Unlike sync version, doesn't support cycles.
- 🆕 **Unstable `moon daemon`** -- background process, file watcher, cache invalidation. New `moon daemon start/stop/...` subcommands; `unstable_daemon` workspace setting. **Note**: Turbo just finished removing its daemon (v2.8.x). Moon is adding one. Different architectural bets.
- Reduced memory footprint across action/project/task graphs (nodes as ints).
- System toolchain moved from WASM plugin to **built-in** (reverses v2.0's WASM-first direction for core toolchains).
- Temporarily brought back `x86_64-apple-darwin` (Intel macOS).
- `MOON_PIPELINE_AUTO_CLEAN_CACHE`, `MOON_PIPELINE_CACHE_LIFETIME`, `MOON_PIPELINE_KILL_PROCESS_THRESHOLD` env var support.

### v2.2.1 (April 16)

- Request retry with exponential backoff (up to 3 retries) via proto for transient errors.
- Async affected tracking another 5-10% faster.
- PowerShell `-EncodedCommand` instead of `-Command` (Windows quoting fix).
- pnpm v10 multi-document lockfiles parsed correctly.
- TypeScript project refs now start with `./` (prep for TS v7 `baseUrl` removal).
- Remote cache pre-check errors now treated as cache misses (correctness).
- `--summary` respected in `moon ci`.

### v2.2.2 (April 20)

- Plugin-loading regression fix (perf regression in graph operations).
- `unpack` extension missing-plugin-locator fix.
- Python toolchain dependency-direction fix (pip/uv).
- VS Code extension pointed at v2 JSON schemas (v1 users must pin older extension).

### v2.2.3 (April 21) -- Same-Day Patch

- **Fixed potential deadlock in action pipeline under many sync heavy operations.** Shipped one day after 2.2.2. Signals a real production bug.

### v2.1.4 (April 4 -- pre-2.2 patch)

- `moon toolchain download` / `moon extension download` to prime plugin caches.
- EPIPE fix in streaming child processes.
- Go: `go list` no longer requires `go.mod`.

### Strategic Implications for Nx

**Threat level: Medium (upgraded from Low-Medium).**

Moon v2.2 is the most substantial Moon release in the scan window and shifts the competitive picture meaningfully:

1. **Direct large-repo performance push** -- 100-170% graph-build speedup + 100-150% affected tracker speedup is material. Matches Turbo's "easier adoption in large repos" narrative from 2.9. Moon is no longer just "technically innovative niche tool"; it's actively targeting Nx/Turbo's large-workspace pain points.
2. **Daemon pivot**. Turbo removed its daemon (v2.8.11); Moon is adding one (v2.2.0). The architectural bets are diverging. Moon's bet: a daemon can cache-invalidate continuously via file watching, amortizing cold-start cost. Turbo's bet: subprocess-to-in-process migration eliminates the need for a daemon.
3. **System toolchain going built-in** (away from WASM) signals moonrepo acknowledging WASM plugin latency for core toolchains. Validates that WASM-first is a narrower bet than v2.0 suggested.
4. **Deadlock patch within 24h of 2.2.2** shows active use at scale and responsive maintenance.
5. **Nx Cloud's remote cache story is more mature** than Moon's (which still treats remote-cache errors as cache misses as of v2.2.1 -- a correctness improvement, not yet cache-hardening).

Moon is still small in market share but the v2.2 performance push plus sustained release cadence (6 releases in 27 days) make it a credible mid-tier competitor, no longer a curiosity. Worth more serious monitoring.

## Bazel

**Latest LTS**: v9.1.0 (April 20, 2026) -- NEW minor
**Patch**: v9.0.2 (April 9, 2026) -- NEW patch
**Prior**: v9.0.1 (March 10), v9.0.0 (Jan 20), v8.6.0 (Feb 26)

### Key Changes in Bazel 9.1.0 (April 20) -- Minor

**Remote Execution** (strategic):

- **Experimental `--rewind_lost_inputs`** -- rerun actions within a single build to recover from remote/disk cache evictions. Addresses a long-standing pain point in large, long-running builds where intermediate artifacts get evicted mid-build.
- **`--experimental_remote_cache_chunking`** -- read/write large blobs to/from remote cache in chunks. Requires server support. Direct parallel to Turbo's v2.9.5 chunk-size bump; both tools are hitting large-blob transfer limits.

**Configurability**:

- `--test_summary=short_uncached` / `detailed_uncached` suppresses cached-test-result reporting.
- Cross-platform launcher fix (windows binary on linux machine).
- `package_group` external repo labels.

**External Deps**:

- `compatibility_level` / `max_compatibility_level` now no-ops -- explicit guidance: module maintainers should provide clear error messages instead.
- Repo contents cache keys now include host OS + CPU arch.
- Remote repo contents cache supports all reproducible repo rules.

**Starlark**: `string.splitlines()` no longer treats U+0085 (NEL) as newline (correctness).

### Key Changes in Bazel 9.0.2 (April 9) -- Patch

- Windows fixes: case-insensitivity in include verification, cross-jlink Windows arm64 builds, `embedded_zipper_sources` windows+unix inclusion.
- `TaskDeduplicator` infinite spin-wait + missing cancellation propagation fix.
- Treat `RUNFILES_TREE_MARKER` as directory in `BuildEvent.forArtifact()`.
- Platform-specific startup bazelrc flags allowed.

### Strategic Implications for Nx

**Threat level: Low for JS/TS (unchanged).** Bazel 9.1.0 is meaningful for Bazel users -- `--rewind_lost_inputs` and remote-cache chunking are real improvements -- but no JS/TS crossover. The chunking flag in Bazel arriving at the same time as Turbo's 256KB chunk size is coincidental convergence on a well-known bottleneck.

## Cross-Competitor Themes

1. **Large-repo performance is the universal theme.** Turbo 2.9 ("easier adoption in large repos", 96% faster, 256KB upload chunks, Rayon boundaries), Moon v2.2 (async graph builder 100-170% faster, async affected 100-150% faster), Bazel 9.1.0 (`--experimental_remote_cache_chunking`, `--rewind_lost_inputs`). Every competitor ships a perf/scale release this cycle.
2. **Remote cache large-blob handling is a shared bottleneck.** Turbo (8KB→256KB chunks) and Bazel (experimental chunking) independently ship similar changes within weeks. Likely a well-known limit being attacked industry-wide.
3. **Architectural divergence on daemons.** Turbo completes daemon removal (v2.8.x); Moon adds a daemon (v2.2). Opposite bets on the same problem (stateful cache invalidation vs in-process startup).
4. **Auth modernization.** Turbo moves to OAuth/device flows for Vercel (canary.11). Gradle caching+CI blogs signal Develocity-side auth/governance work. Convergence toward low-friction first-connection UX.
5. **AI docs hardening continues.** Turbo factored `ai-agent-detection` out into `@vercel/agent-readability` (v2.9.4) -- Vercel turning AI-readable docs into a reusable npm package, extending the moat beyond Turbo's own docs.
6. **3.0 on Turbo's horizon.** First public signal. Nx should assume a Turbo 3.0 release within 2026.
