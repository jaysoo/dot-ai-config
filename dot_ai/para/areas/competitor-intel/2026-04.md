## Turborepo

**Latest stable**: v2.9.4 (April 5, 2026) -- npm `turbo@latest` = 2.9.4 (verified)

### Turborepo 2.9.0 -- Major Release (March 30)

**Up to 96% Faster Time to First Task** (headline):

- Vercel backend monorepo (1037 packages): 8.1s -> 716ms (91% faster)
- Frontend repo (132 packages): 1.9s -> 361ms (81% faster)
- create-turbo (small): 676ms -> 132ms (80% faster)
- Achieved via: replacing O(V^3) Floyd-Warshall with O(V+E) DFS in watch mode subgraph, batching per-package git subprocess calls into repo-wide index, parallelizing RepoGitIndex, optimizing hash computation (avoid clones, unstable sorts), replacing twox-hash with xxhash-rust, reducing globwalk allocations, deferred TLS initialization to background thread

**`turbo query` Graduated to Stable**:

- GraphQL-based workspace introspection API is now the canonical interface
- New shorthands: `turbo query affected` (replaces `turbo-ignore`), `turbo query ls` (replaces `turbo ls`)
- `turbo-ignore` officially deprecated with tailored migration notice for Vercel users
- `turbo scan` deprecated -- consolidated into query engine
- **Everything is a query** -- unified intelligence API surface

**Adoption Improvements**:

- Circular package dependencies no longer block Turborepo adoption
- Task Graph validation replaces Package Graph validation -- task graph cycles are impossible when tasks lack dependencies
- Easier incremental adoption in large repos with existing circular deps

**OpenTelemetry Observability (experimental)**:

- `experimentalObservability` config sends build metrics to OTLP-compatible backends (Grafana, Datadog)
- Pre-configured Grafana dashboard example (`with-otel`)
- Delta temporality for metrics, `runAttributes` config for cardinality control

**Structured Logging**:

- `--json` flag for machine-readable newline-delimited JSON output
- `--log-file` flag for file-based structured logging
- Infrastructure for CI dashboard integration and AI agent consumption

**LLM-Friendly Profile Output**:

- `--profile` now generates LLM-friendly markdown alongside Chrome trace output
- Optional filename for `--profile` and `--anon-profile`
- **Strategic**: Build trace data formatted for AI agent consumption

**Deprecations for 3.0**:

- `turbo-ignore`, `turbo scan`, `--parallel`, `--no-cache`, `TURBO_REMOTE_ONLY`, daemon-related flags, graphviz-dependent graph output formats all deprecated
- Future Flags pattern for gradual migration: `affectedUsingTaskInputs`, `filterUsingTasks`, `globalConfiguration`, `longerSignatureKey`
- Closed 141 issues (305 reactions), reduced open issues by 70%

### v2.9.1 -- v2.9.4 (March 30 -- April 5)

**Incremental Task Caching** (v2.9.4 -- strategic):

- New `feat: Add incremental task caching` -- tasks can incrementally cache outputs
- This is a significant new caching capability, directly relevant to Nx's caching story

**Local Cache Eviction Controls** (v2.9.1):

- `cacheMaxAge` and `cacheMaxSize` config options for automatic local cache eviction
- Addresses long-standing user complaint about unbounded cache growth

**Combined `--affected` and `--filter`** (v2.9.4):

- `--affected` and `--filter` can now be used together
- Enables: "run only affected tasks in this filtered subset of packages"

**Task Graph Filtering Refinements** (v2.9.2, v2.9.4):

- Include transitive dependencies in engine graph pruning for affected paths using Task Graph
- Prevent `filterUsingTasks` `--filter` from pulling dependents into Task Graph
- Retain microfrontend proxy tasks when using `filterUsingTasks`

**AI/Agent Integration** (v2.9.0, v2.9.4):

- AI agent detection via `@vercel/agent-readability` (replaces local implementation)
- AI-generated response disclaimer updated to include human attribution
- Docs site continues to auto-detect and rewrite for AI agents

**Lockfile & Pruning Hardening** (v2.9.1-v2.9.4):

- Package manager catalog support in migration codemods
- Bun workspace lockfile pruning fix
- pnpm workspace importer entry backfill during prune
- npm shallow install strategy preserved during prune
- pnpm per-workspace lockfile preservation
- Respect dirty `.gitignore` patterns during task input hashing

**Agentic Repo Automation** (v2.9.0 internal):

- Internal "agents app" for repo automation with dashboard, run tracking, live logs
- Uses AI agents for security audit automation and issue triage
- **Turborepo team using AI agents to maintain their own repo** -- dogfooding agentic workflows

### What's Coming (v2.9.5 canary signals -- 5 canaries April 6)

- Specific features TBD from canary release notes

### Strategic Implications for Nx

**Threat level: HIGH (upgraded from Medium-High).**

The v2.9.0 release marks Turborepo's transition from "catching up" to "competing credibly" across multiple dimensions simultaneously:

1. **Performance narrative is now compelling.** 96% faster Time to First Task with real benchmarks (1037-package repo). Nx must counter with its own benchmarks, particularly emphasizing end-to-end CI time (where Nx Cloud's distribution is the differentiator, not startup time).

2. **Incremental task caching is new territory.** v2.9.4's incremental caching is a capability Nx doesn't have in the same form. This could be significant for long-running tasks (e2e tests, builds). Monitor closely for what "incremental" means in practice.

3. **`turbo query` is now stable and canonical.** The "everything is a query" consolidation is complete. Combined with `turbo-ignore` deprecation, Vercel users are now forced to use the query engine. This creates a captive audience for Turborepo's workspace intelligence API.

4. **Local cache management is a real user need.** `cacheMaxAge`/`cacheMaxSize` addresses a pain point that Nx hasn't solved cleanly. Consider implementing similar cache eviction controls.

5. **Combined `--affected` + `--filter` closes another gap.** Users can now express "run affected tasks in this subset" -- a pattern Nx supports via the project graph but Turborepo lacked.

6. **AI agent strategy is maturing.** LLM-friendly profile output + `@vercel/agent-readability` for docs + internal agentic repo automation = Turborepo's AI strategy is moving from marketing to substance.

7. **Release cadence remains aggressive.** 5 stable v2.9.x releases in 7 days, plus canaries. The velocity is real and sustained.

**Nx should consider**:

- **Publish benchmark comparisons** for CI scenarios (not just CLI startup) where Nx Cloud distribution dominates
- **Evaluate incremental caching** for tasks like e2e tests -- this could be a compelling feature
- **Ship local cache eviction controls** -- cacheMaxAge/cacheMaxSize equivalents
- **Ensure AI-friendly doc endpoints** are on the roadmap -- Turborepo is now clearly ahead here
- **Continue emphasizing unassailable features**: `nx migrate`, `nx import`, `nx release`, module federation, CIPE insights

## Moon

### Moon v2.0.0 "Phobos" (Feb 18) -- Major Release

**WASM Plugin Architecture** (headline):

- Replaced legacy platform system with WASM-based plugin system
- Community-contributed language toolchains now possible
- Plugin APIs: `define_extension_config`, `extend_project_graph`, `extend_task_command`, `extend_task_script`, `sync_project`, `sync_workspace`
- Stabilized what was previously "unstable" toolchains

**Multi-Format Configuration**:

- JSON, JSONC, HCL, Pkl, TOML, YAML all supported (was YAML-only)
- `.config/moon` alternative to `.moon` directory

**Remote Caching Stabilized**:

- All remote caching settings promoted to stable
- gzip/zstd compression for HTTP requests
- (Note: correctness issues patched in v2.1.1 -- batching max size/limit bugs)

**MCP Integration**:

- Updated to MCP protocol 2025-11-25
- Added `generate` tool for code generation via MCP
- `get_projects` and `get_tasks` return fragments to reduce payload size

**Task System Rework**:

- Deep merge for task inheritance (was shallow merge)
- `command`/`args` only support simple commands; `script` for compound commands
- Reworked env var merge order and substitution
- `.env` loading moved to before task execution (was creation time)
- New `utility` task preset

**Migration Extensions**:

- **Migrate Nx** extension: supports `targets.*.continuous` from project.json
- **Migrate Turborepo** extension: supports `tags`, `tasks.*.env` (wildcards and negation)
- **Both extensions directly target Nx and Turborepo users for migration**

**Docker**: Better toolchain integration, `--no-setup` and `--template` support, per-project overrides

**VCS**: New Git v2 implementation replacing experimental version, worktree and submodule support

**CLI Overhaul**: Renamed commands, `moon exec` (low-level execution), `moonx` binary stabilized, `...` alias for `**/*` (Bazel-style targets)

### Moon v2.1.0 -- v2.1.4 (March 16 -- April 4)

**Execution Plans** (v2.1.0):

- Declarative JSON files configuring targets, affected filtering, graph traversal, and pipeline behavior
- Dependency targeting by scope: `^production:build` syntax
- Higher-level orchestration abstractions without Nx equivalents

**Affected Files Enhancements** (v2.1.0):

- `affectedFiles.filter` -- glob patterns to filter affected files before passing to tasks
- `affectedFiles.ignoreProjectBoundary` -- include workspace-relative files beyond project boundary
- `affectedFiles.passDotWhenNoResults` -- fallback behavior control
- More granular than Nx's affected analysis at the task-input level

**On-Demand Affected Tracking** (v2.1.2):

- Affected tracker marks project/tasks as affected on-demand instead of up front
- Short-circuits early for large repos
- Projects/tasks loaded without expansion -- reduces memory and improves startup

**TypeScript v6 Support** (v2.1.2)

**Go Toolchain Enhancements** (v2.1.0-v2.1.3):

- `go list --deps` for project relationship discovery
- `inferRelationships` and `inferRelationshipsFromTests` settings for user control
- Fixed `go list` not running in project root

**Python**: PEP 503 normalization, venv fixes, dependency inference with extras metadata

**Rust**: Updated Cargo.lock parsing

**Task System**: `runInSyncPhase` (run tasks during `moon sync`), `pruneProjectReferences` for TypeScript

**v2.1.4** (April 4): Bug fix release -- details TBD (release notes minimal)

### Strategic Implications for Nx

**Threat level: Low-Medium (unchanged).** Moon v2.0 is architecturally ambitious and the migration extensions explicitly target Nx users. However:

- Market share remains small; v2.0 migration is painful (many breaking changes)
- The WASM plugin system is technically innovative but ecosystem is nascent
- Affected files granularity (v2.1.0) and on-demand tracking (v2.1.2) are technically ahead of Nx in some respects
- Migration extensions for Nx and Turborepo show Moon positioning as an "upgrade from" both tools
- `runInSyncPhase` is a lifecycle concept without Nx equivalent
- Rapid release cadence (10 releases in the window) shows active maintenance

## Cross-Competitor Themes

### 1. Performance as Table Stakes

Every competitor invested in performance this cycle:
- Turborepo: 96% faster TTFT via algorithmic + systems optimization
- Moon: On-demand affected tracking, lazy project/task loading
- Pants: `frozendict` ported to Rust
- All converging on "startup should be sub-second" as the baseline expectation

### 2. AI/Agent Integration Deepening

- **Turborepo**: LLM-friendly profile markdown, `@vercel/agent-readability` for docs, internal agentic repo automation, AI agent detection in wire protocol
- **Moon**: MCP v2025-11-25 protocol, `generate` MCP tool
- **Gradle/Develocity**: 2 MCP servers, AI failure grouping, natural-language querying
- **Pants**: AI tooling contribution notice
- Industry converging on MCP as the standard interface, structured output for agent consumption, and AI-assisted build intelligence

### 3. Query/Intelligence APIs Emerging

- **Turborepo**: `turbo query` now stable -- GraphQL-based unified intelligence API
- **Moon**: Execution plans as declarative JSON
- **Bazel**: `bazel mod show_repo` with visibility flags
- Build tools moving from "run tasks" to "understand the workspace programmatically"

### 4. Remote Caching Is Table Stakes, Cache Intelligence Is the New Frontier

- Turborepo: Incremental task caching, local cache eviction controls, cache restore optimization
- Moon: Remote caching stabilized (still patching correctness)
- Bazel: Mature remote execution
- **The frontier is now**: cache provenance, cache debugging, incremental caching, smart eviction

### 5. CI-Specific Configuration Emerging

- Turborepo: `experimentalCI` task config key (tasks behave differently in CI)
- Gradle: Three blog posts on GitHub Actions integration
- **Local vs CI behavior differentiation is a growing theme** -- directly relevant to Nx Cloud

### 6. Migration Tooling as Competitive Weapon

- Moon v2.0: Explicit "Migrate Nx" and "Migrate Turborepo" extensions
- Turborepo: Package manager catalog support in migration codemods
- Bazel: Ecosystem pressure from CcInfo removal requiring rule upgrades
- **Migration is a competitive vector** -- Nx's `nx migrate` remains unmatched

## Recommended Actions

### Immediate (This Sprint)

1. **Evaluate Turborepo's incremental task caching** (v2.9.4). Understand what it actually does (scope, limitations) and whether Nx should offer something similar. This is the most significant new capability.

2. **Ship local cache eviction controls.** `cacheMaxAge`/`cacheMaxSize` equivalent. Common user pain point, now solved by Turborepo. Quick win.

3. **Publish Nx vs Turborepo 2.9 benchmarks** focusing on CI scenarios (task distribution, remote cache hit rates, e2e pipeline time) where Nx Cloud dominates -- not just CLI startup time where Turborepo now has a strong narrative.

### Near-Term (This Quarter)

4. **AI-friendly documentation endpoints.** Turborepo has `.md` routes, auto-rewrite for agents, LLM-friendly profile output. Nx is behind on all three. At minimum: markdown doc routes and structured sitemap.

5. **Monitor `experimentalCI` graduation.** If this becomes "turbo.json tasks automatically behave differently on Vercel CI," it's a significant competitive move against Nx Cloud's CI optimization surface.

6. **Counter Moon's migration extensions.** Moon v2.0 ships explicit "Migrate Nx" and "Migrate Turborepo" extensions. While Moon's market share is small, this is a signal that competitors are investing in migration-from-Nx tooling.

### Medium-Term (This Half)

7. **Evaluate whether Nx should expose a unified query/intelligence API** similar to `turbo query`. The "everything is a query" consolidation is architecturally clean and increasingly useful for AI agent consumption.

8. **Watch Turborepo's OTel path.** When this becomes "Vercel Build Insights" in the Vercel dashboard, it will be a direct Nx Cloud competitor for build analytics. Nx Cloud's depth (CIPE, cache debugging, task distribution analytics) should be the counter-narrative.
