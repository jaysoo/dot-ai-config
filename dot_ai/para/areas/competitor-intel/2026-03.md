## Turborepo

**Latest stable**: v2.8.13 (March 3, 2026) -- npm `turbo@latest` = 2.8.13
**Previous stable**: v2.8.12 (Feb 27), v2.8.11 (Feb 25)
**Major feature release**: v2.8 blog post (January 23, 2026)

### Key Changes in v2.8.13 (March 3)

**OpenTelemetry Observability** (new feature):

- `experimentalObservability` config added to `@turbo/repository` library with OTel backend
- Ships with a pre-configured Grafana dashboard example (`with-otel`)
- OTel metrics sent without requiring `--summarize` flag
- This is Turborepo's first step into build observability/analytics -- territory where Nx Cloud has been operating

**Performance** (significant push this release):

- **Replaced SWC with Oxc** for JS/TS parsing -- aligns with VoidZero ecosystem
- Stream file contents during hashing to lower memory usage
- Eliminated per-directory stat syscalls in untracked file detection
- Replaced `git` subprocess in worktree detection with pure Rust
- Optimized npm lockfile parser
- Reduced allocations in file hash computation
- Disabled unused zstd default features to reduce C compilation time

**Lockfile Pruning Fixes** (ongoing focus area):

- Exclude peer deps from workspace external dep resolution
- Treat `npm:` alias dependencies as external, not workspace references
- Preserve `file:` and `link:` protocol entries in pruned bun lockfiles
- Retain injected workspace package entries during pnpm lockfile pruning
- Filter orphaned Yarn packageExtensions entries during pruning
- Support pnpm per-workspace lockfiles in `turbo prune`

**Other**:

- `turbo dev` no longer hangs when daemon file watching fails
- Microfrontends: merges `with` into root config instead of replacing it
- Framework inference now checks `devDependencies` (not just `dependencies`)
- 100% of prysk tests now ported to pure Rust

### Key Changes in v2.8.11 (Feb 25)

**Daemon Removed from `turbo run`** -- daemon flags and config deprecated. This simplifies the architecture.

**Massive Rust Performance Sprint** (~25 perf commits in a single release):

- Deduplicated file hashing and parallelized globwalks
- Improved transitive dependency resolution cache sharing across workspaces
- Reduced filesystem syscalls in globwalk, SCM hashing, and task scheduling
- Parallelized package.json loading, lockfile parsing, task hash computation
- Replaced `libgit2` with `gix-index` for faster file hashing
- Parallelized `turbo run` pre-execution hot path
- Used `Arc<str>` for task dependency hashes to avoid heap clones
- Fast path for shallow wildcard glob patterns in workspace discovery

### Key Changes in v2.8.12 (Feb 27)

- Consolidated duplicate Rust dependencies (thiserror, itertools, petgraph, dashmap)
- Updated axum 0.7 to 0.8, streamlined tower/tower-http
- Removed libgit2/git2 dependency, replaced with gix-object
- Removed async-graphql from turborepo-lib
- Extracted query module into turborepo-query crate
- Fixed bun lockfile pruning issues with peerDependencies and nested packages
- Added test retries for flaky tests

### Key Changes in v2.8 (shipped Jan 23)

- **Git worktree support**: Shared local cache across worktrees
- **Agent Skill** (agentskills.io): Makes coding agents "Turborepo experts"
- **AI-friendly documentation**: `.md` routes, `/sitemap.md`, versioned doc subdomains
- **`turbo docs` subcommand**: Terminal-based doc search
- **Task descriptions**: New `description` field in turbo.json
- 18 features, 50 fixes, 14 doc improvements total

### What's Coming (v2.8.14 canary signals)

- **OTel refinements**: Delta temporality for metrics export, `runAttributes` config for cardinality control -- shows continued investment in observability
- **Security**: `futureFlags.longerSignatureKey` to enforce minimum HMAC key length for cache signing
- **Bun**: Treat Bun runtime modules as builtins in Boundaries feature
- **Perf**: Skip irrelevant packages when using `package#task` syntax

### Strategic Implications for Nx

**Threat level: Medium-High on narrative, Medium on substance (upgrading from Low).**

The v2.8.11-v2.8.13 cycle represents a genuine engineering acceleration for Turborepo:

1. **Performance is now a real differentiator.** The sheer volume of Rust performance work (~40+ perf commits across 3 releases) shows Turborepo is investing heavily in raw speed. Parallelized hashing, eliminated git subprocesses, Oxc replacing SWC -- these are substantive improvements, not marketing.

2. **OpenTelemetry observability is a strategic move.** This is Turborepo's first step into build analytics/observability, which has been Nx Cloud territory. Today it's "experimental" with a Grafana dashboard. Tomorrow it could be "Vercel Build Insights" -- a direct Nx Cloud competitor feature.

3. **Daemon removal simplifies the architecture.** Less surface area = fewer bugs = better DX. Nx should note this simplification trend.

4. **Oxc adoption signals VoidZero alignment.** Turborepo replacing SWC with Oxc (VoidZero's parser) shows Vercel and VoidZero ecosystems converging. This could mean deeper Vite+/Turborepo integration down the road.

5. **AI narrative continues.** Agent Skills, `.md` doc routes, "agentic coding" positioning remain marketing differentiators.

**Nx should consider**:

- Publishing agent-friendly doc formats (`.md` routes, structured sitemap)
- Monitoring Turborepo's OTel observability path -- if Vercel builds a Build Insights product on top, it competes with Nx Cloud analytics
- Emphasizing Nx's file-level affected analysis vs Turborepo's package-level approach as the performance story shifts

## Moon

**Latest stable**: v2.0.3 (February 26, 2026 -- re-release of v2.0.2 after distribution workflow failure)
**Major release**: v2.0.0 "Phobos" (February 18, 2026)

### Key Changes in v2.0.0

**WASM Plugin Toolchain System** (biggest change):

- Replaced hard-coded platform system with WASM plugins for toolchains
- Plugins can extend project graphs, modify task commands, integrate with Docker, parse dependency manifests, manage tool installations, influence task hashing
- Community can create custom toolchains for any language
- Tasks can now associate with multiple toolchains (previously limited to one platform)
- New `.moon/extensions.*` config file for extension plugins

**CLI Overhaul**:

- New commands: `moon exec` (low-level executor), `moon projects`, `moon tasks`, `moon hash`, `moon extension`
- Interactive selection when commands lack required identifiers
- All CLI options standardized to kebab-case

**Configuration Flexibility**:

- Multi-format support: YAML, JSON, JSONC, HCL, Pkl, TOML
- Optional `.config/moon` directory structure (alternative to `.moon`)
- Config file renamed: `.moon/toolchain.yml` -> `.moon/toolchains.yml` (plural)

**Task System**:

- Explicit `inheritedBy` conditions (toolchains, stacks, layers, languages, tags, file presence)
- New `utility` preset disables caching/CI for local-only tasks
- Deep merging replaces shallow for task inheritance
- Command/args restricted to simple commands; compound ops require `script` field

**Remote Caching**: Stabilized with compression support

**Docker**: Workspace-level defaults with per-project overrides; Tera-based Dockerfile templates

**VCS**: Unified Git implementation on faster v2 variant with worktree compatibility

### Breaking Changes

Extensive. Too many to list per their own docs. Key ones:

- `moon upgrade` no longer works; must reinstall via shell scripts
- Stricter command parsing (no pipes/redirects in command field)
- Deferred env file loading until execution time
- Config file renames and restructuring

### Patch Releases

- v2.0.1 (Feb 20): Fixed upgrade command for new distribution format, WASM serialization fixes
- v2.0.3 (Feb 26): Re-release of v2.0.2 (workflow failure). Temporarily relaxed shallow checkout errors in CI, added Docker prune logging, added `MOON_INCLUDE_RELATIONS` env var, fixed graph visualizer rendering in VS Code extension, fixed `shell: false` being force-enabled on glob/env detection, fixed CI run checks for "run" type tasks, added `.env`/`.env.*` to default `hasher.ignoreMissingPatterns`

### Strategic Implications for Nx

**Threat level: Low-Medium.** Moon's WASM plugin architecture is technically impressive and validates the plugin-first approach Nx pioneered. However:

- Moon has very small market share compared to Nx or Turborepo
- The v2.0 upgrade path is painful (many breaking changes, reinstall required)
- Moon targets polyglot repos more than JS/TS-first teams
- Multi-format config (HCL, Pkl, TOML) is a nice flexibility play but niche

**What validates Nx**: Plugin-based extensibility, remote caching, Docker integration, task graph intelligence -- all areas where Nx has mature offerings.

**What Nx should watch**: WASM-based plugin architecture for community extensibility. Moon's approach of letting the community build toolchain plugins could inspire Nx plugin improvements. The multi-config-format support is also a UX differentiator worth noting.

## Bazel

**Latest LTS**: v9.0.0 (January 20, 2026)
**Latest patch**: v8.6.0 (February 26, 2026, on the 8.x LTS track)

### Key Changes in Bazel 9.0 LTS

**WORKSPACE Fully Removed**:

- Bzlmod is now the only dependency management system
- All WORKSPACE code deleted from Bazel core
- Migration tool provided; the team acknowledges "migration is hard"
- Enables new capabilities: local + experimental remote repository contents caching

**Starlarkification Complete**:

- All C++ rules moved to external `rules_cc` module
- `--incompatible_autoload_externally` defaults to empty string
- Language-specific rules must be explicitly loaded from respective modules
- Allows rules to evolve independently of Bazel releases

**Prebuilt Protobuf Compiler**: Protobuf 33.4+ supports prebuilt protoc, eliminating unnecessary rebuilds

**Performance**: Periodic garbage collection of persistent action cache while server is idle

**CLI**: `bazel --quiet` startup option; `executables()` function in query commands

### Key Changes in Bazel 8.6.0 (Feb 26)

- Suppress cached test result reporting with `--test_summary=short_uncached`
- `ctx.actions.symlink` accepts `target_type` argument
- Remote downloader TLS handshake failover to mirrors
- `bazel mod show_repo` new flags: `--all_repos`, `--all_visible_repos`
- Overlaid files in http_archive now executable

### Strategic Implications for Nx

**Threat level: Low for JS/TS ecosystem.** Bazel continues to serve enterprise-scale polyglot monorepos (Google, Stripe, etc.). The WORKSPACE removal is a decade-long cleanup that shows Bazel's maturation but doesn't change its positioning relative to Nx:

- Bazel's complexity and learning curve remain prohibitive for most JS/TS teams
- The Starlarkification completion actually increases setup complexity (more explicit imports)
- Bazel's strengths (hermetic builds, massive scale) are overkill for typical Nx users

**What to note**: Bazel's investment in migration tooling and the "agent-driven setup" mention for Bzlmod migration shows even Bazel is thinking about AI-assisted onboarding.

## Vite+ (VoidZero)

**Latest**: `0.0.2-g17a37daf.20260304-1136` (daily builds, up from `0.0.0-gXXXX` versioning)
**Status**: Technical preview / early adopter. No stable release. Evan You speaking at Vue.js Amsterdam March 13 with "exciting announcements."
**Funding**: $17.1M (Accel-led Series A, Oct 2025)

### What It Is

Vite+ is a source-available, commercially-licensed superset of Vite built by Evan You's company VoidZero. It bundles Rolldown (Rust bundler), Oxc (Rust parser/linter/formatter), and Vitest into a single `vite` CLI with added commands: `vite run` (task runner with caching), `vite test`, `vite lint`, `vite fmt`, `vite lib`, `vite new`, `vite ui`.

Key team: Evan You, Boshen Chen (Oxc creator), Christoph Nakazawa (Jest/Yarn/Metro creator).

### Ecosystem Updates (February 2026)

**Oxfmt reached beta** -- 100% Prettier JS/TS conformance tests passing, up to 36x faster than Prettier. Notable adopters: **Turborepo**, Hugging Face, Lichess, Oxide Computer. The Turborepo adoption is significant -- it signals Vercel/VoidZero ecosystem convergence.

**Vitest 4.1 beta**: `aroundEach` and `aroundAll` hooks for context wrapping and tracing. Upcoming UI filtering for slow tests.

**Rolldown**: Up to 9.6% faster via internal file path optimization. tsdown added experimental Node.js single executable bundling (`--exe`).

**Oxc/Oxlint**: Semantic analysis 4-6% faster. Type-aware linting config option. Now supports 59 of 61 `typescript-eslint` rules via tsgolint. New `oxc-webpack-loader` for webpack-to-Rolldown migration.

**Vite 8 beta**: Experimental devtools for module graph inspection and HMR debugging. New plugin registry for npm-published Vite plugins.

### Key Competitive Feature: `vite run` (Monorepo Task Runner)

- Built-in task runner with "intelligent caching" and "smart input inference" -- claims zero-config cache invalidation
- Works with arbitrary tasks, not just Vite+ commands
- Positioned as Turborepo/Nx alternative: "like Turborepo but without telling the system how to invalidate the cache"
- **No remote caching, no affected commands, no distributed CI, no project graph analysis** -- only local caching

### What They Have That We Don't

- Unified Rust-powered toolchain (build+test+lint+format in one CLI)
- 40x faster builds (Rolldown), 100x faster linting (Oxlint)
- Integrated formatting (Oxfmt -- now in beta, 36x faster than Prettier)
- Supply chain security / dependency vetting (enterprise tier)
- "Zero-config" caching narrative
- `oxc-webpack-loader` for webpack migration path (could pull webpack users away from Nx's migration story)

### What We Have That They Don't

- Remote caching (Nx Replay), distributed CI (Nx Agents), affected commands
- Polyglot support (Java, Go, .NET), Angular first-class support
- Migration generators, code generators, workspace conformance, CODEOWNERS
- Module federation, mature plugin ecosystem
- MIT license (theirs is source-available/commercial)

### Companion Potential

**Yes.** Nx can orchestrate Vite+ commands while adding remote caching + distribution + affected. "Use Vite+ for speed, Nx for scale." Tension point: if they add remote caching, that directly competes with Nx Cloud.

### Strategic Implications

**Threat level: Medium-High (long-term).** Not critical today (no stable release, no remote cache, no affected). But:

- Versioning moved from `0.0.0` to `0.0.2` -- indicating progression toward a release
- Vue.js Amsterdam talk March 13 may include stable release announcement or public preview
- Oxfmt beta + Turborepo adoption shows the VoidZero ecosystem is gaining real traction
- `oxc-webpack-loader` for webpack migration is a wedge that could pull users toward the Vite+ ecosystem

Biggest risk remains **narrative capture** -- if "Vite+" becomes the default answer for new Vite monorepos, Nx loses top-of-funnel.

## Strategic Implications for Nx

### Immediate Actions

1. **Monitor Turborepo's OTel observability path.** The `experimentalObservability` feature in v2.8.13 is Turborepo's first entry into build analytics. If Vercel builds a "Build Insights" product on top (integrating with their dashboard), it directly competes with Nx Cloud's task analytics and CIPE insights. Nx Cloud should emphasize its existing depth here.

2. **Counter Turborepo's "agentic coding" narrative.** Agent Skills, `.md` doc routes, and "agentic coding" positioning remain a marketing gap. Nx should:
   - Ship agent-friendly documentation endpoints (`.md` routes, structured sitemap for agents)
   - Position Nx Cloud's intelligence features (affected detection, task distribution) as the deeper AI story
   - Emphasize file-level affected analysis vs Turborepo's package-level approach

3. **Track the Vercel-VoidZero convergence.** Turborepo replacing SWC with Oxc, and Turborepo adopting Oxfmt, signals deepening alignment between Vercel and VoidZero ecosystems. If Vite+ and Turborepo integrate more tightly, that creates a stronger combined threat.

### Medium-Term Considerations

4. **Watch Vue.js Amsterdam (March 13).** Evan You's "exciting announcements" could include a Vite+ public preview or stable release. If it includes remote caching or deeper monorepo features, threat level escalates.

5. **WASM plugin extensibility.** Moon's WASM plugin architecture for community-built toolchains is worth studying. If Nx's plugin system could support WASM-based community plugins, it would open extensibility to non-JS ecosystems without core changes.

6. **AI-powered CI troubleshooting.** Gradle's Develocity is shipping AI analysis of CI failures. Nx Cloud is well-positioned to offer similar intelligence (why did this CI run fail? what changed?). This is a natural extension of the build intelligence platform vision.

### Validation of Nx's Approach
