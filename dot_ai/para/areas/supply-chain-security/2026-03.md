## CI Workflow Security

### Action Version Pinning

**Status: GOOD (with one exception)**

All workflow files use SHA-pinned action versions with version comments. Examples:

- `actions/checkout@08c6903cd8c0fde910a37f88322edcfb5dd907a8 # v5.0.0`
- `actions/setup-node@a0853c24544627f65ddf259abe73b1d18a591444 # v5.0.0`
- `actions/cache@0400d5f644dc74513175e3cd8d07132dd4860809 # v4.2.4`

**Exception**: `banner-monitor.yml` uses unpinned tag references:

- `actions/cache/restore@v4` (line 39)
- `actions/cache/save@v4` (line 89)

Low risk (scheduled workflow, non-sensitive content) but should be pinned for consistency.

### Trigger Security

- **No `pull_request_target` triggers** -- eliminates primary vector for secret exfiltration from fork PRs
- CI uses `pull_request` trigger (safe -- runs in fork context without secrets)
- Publish workflow triggers: `schedule`, `workflow_dispatch`, `release` (all safe)

### Secret Handling

- `NX_CLOUD_ACCESS_TOKEN`: Used in CI, scoped to the CI environment
- `NETLIFY_AUTH_TOKEN`: Used only in `banner-monitor.yml` (scheduled, no PR trigger)
- `SLACK_WEBHOOK_URL`: Used in notification-only jobs
- `GITHUB_TOKEN`: Default token with scoped permissions
- No hardcoded tokens or secrets in workflow files

### Script Injection Prevention

- PR title/body passed as **environment variables** (not inline `${{ }}`), preventing injection
- `do-not-merge.yml` uses `toJSON()` for label data (safe JSON serialization)
- No direct string interpolation of untrusted user input in `run:` steps
