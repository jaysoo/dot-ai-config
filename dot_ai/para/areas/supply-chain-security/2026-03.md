# Supply Chain Security Audit -- March 2026

_Scan date: 2026-03-02_

## Summary

**Overall status: WARNING**

No critical vulnerabilities in direct Nx packages. The S1ngularity incident (August 2025) has been fully remediated with 2FA enforcement and npm Trusted Publishers. CI workflow security is strong overall, with one minor finding (unpinned cache actions in `banner-monitor.yml`). The monorepo's dev dependency tree carries 197 known vulnerabilities (0 critical, 85 high, 68 moderate, 44 low), most from transitive dev dependencies not shipped to end users.

- **npm provenance attestations**: PRESENT on `nx`, `@nx/devkit`, `@nx/js` (and likely all packages)

- **2FA**: Enforced on all npm packages post-S1ngularity (cannot publish with tokens alone)
- **Trusted Publishers**: Implemented via OIDC (`id-token: write` in publish workflow)

### Post-S1ngularity Remediation (CVE-2025-10894)

The Nx team implemented comprehensive security measures after the August 2025 supply chain attack:

1. All npm packages require 2FA -- cannot be published with access tokens
2. npm Trusted Publishers via OIDC authentication (GitHub Actions to npm)
3. PR release script verification (compares `scripts/nx-release.ts` and `scripts/publish-resolve-data.js` against master)
4. GitHub environment protection (`npm-registry`) on the publish job
5. Manual approval required for all releases (Slack notification to release manager)

## CI Workflow Security

### Action Version Pinning

**Status: GOOD (with one exception)**

All workflow files use SHA-pinned action versions with version comments. Examples:
- `actions/checkout@08c6903cd8c0fde910a37f88322edcfb5dd907a8 # v5.0.0`
- `actions/setup-node@a0853c24544627f65ddf259abe73b1d18a591444 # v5.0.0`
- `actions/cache@0400d5f644dc74513175e3cd8d07132dd4860809 # v4.2.4`

**Exception**: `banner-monitor.yml` uses unpinned tag references:
- `actions/cache/restore@v4` (line 39)
- `actions/cache/save@v4` (line 89)

This is low risk since the workflow runs only on `nrwl`-owned repos and handles non-sensitive banner content, but should be pinned to SHA for consistency.

### Trigger Security

- **No `pull_request_target` triggers** -- eliminates the primary vector for secret exfiltration from fork PRs
- CI uses `pull_request` trigger (safe -- runs in fork context without secrets)
- Publish workflow triggers: `schedule`, `workflow_dispatch`, `release` (all safe)

### Secret Handling

- `NX_CLOUD_ACCESS_TOKEN`: Used in CI, scoped to the CI environment
- `NETLIFY_AUTH_TOKEN`: Used only in `banner-monitor.yml` (scheduled, no PR trigger)
- `SLACK_WEBHOOK_URL`: Used in notification-only jobs
- `GITHUB_TOKEN`: Default token with scoped permissions
- No `ACTIONS_RUNNER_DEBUG` or `ACTIONS_STEP_DEBUG` exposed
- No hardcoded tokens or secrets in workflow files

### Script Injection Prevention

- PR title/body passed as **environment variables** (not inline `${{ }}`), preventing injection
- `do-not-merge.yml` uses `toJSON()` for label data, which safely serializes to JSON literals
- No direct string interpolation of untrusted user input in `run:` steps

### Permissions (Least Privilege)

| Workflow | Top-level permissions |
|----------|---------------------|
| `npm-audit.yml` | `{}` (none), job-level `contents: read` |
| `e2e-matrix.yml` | `{}` (none), job-level `contents: read` |
| `schedule-stale.yml` | `{}` (none), job-level scoped |
| `issue-notifier.yml` | `issues: read`, `pull-requests: read` |
| `lock-threads.yml` | `issues: write`, `pull-requests: write` |
| `pr-title-validation.yml` | `read-all` |
| `ci.yml` | No top-level restriction |
| `publish.yml` | No top-level restriction (job-level: `id-token: write`, `contents: write`, `pull-requests: write`) |
| `codeql-*.yml` | No top-level restriction (job-level scoped) |

**Note**: `ci.yml`, `publish.yml`, `codeql-*.yml`, and `generate-embeddings.yml` lack top-level `permissions:` declarations. While jobs within these workflows scope their permissions appropriately, adding a restrictive top-level default (`permissions: {}` or `permissions: read-all`) would be a defense-in-depth improvement.

## Recent CVEs

### CVE-2025-10894 (S1ngularity) -- RESOLVED

- **Date**: August 26, 2025
- **Severity**: Critical
- **Advisory**: [GHSA-cxm3-wv7p-598c](https://github.com/nrwl/nx/security/advisories/GHSA-cxm3-wv7p-598c)
- **Impact**: Malicious `postinstall` scripts in compromised Nx packages exfiltrated SSH keys, npm tokens, GitHub tokens, crypto wallets, and environment secrets
- **Duration**: ~4 hours before all malicious versions were removed
- **Affected versions**: Various versions of `nx`, `@nrwl/nx`, `@nx/devkit`, and other packages
- **Current status**: Fully remediated. All malicious versions removed from npm registry. 2FA and Trusted Publishers enforced.
- **Postmortem**: [S1ngularity - What Happened, How We Responded, What We Learned](https://nx.dev/blog/s1ngularity-postmortem)

No additional CVEs affecting Nx packages found for 2026.

## Recommendations
