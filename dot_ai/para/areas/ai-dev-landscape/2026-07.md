## 5. AI Security Threats

### HIGH: Miasma Supply Chain Attack (June 1-17, 2026)

- **32 packages** under `@redhat-cloud-services` npm namespace compromised; **96 malicious versions** published in two waves on June 1.
- Root cause: attacker compromised a GitHub account via a VS Code extension containing malware; used that account to inject code into packages and abused GitHub Actions OIDC trusted-publishing to mint short-lived npm tokens.
- **SLSA provenance attestation forgery for the third time in this campaign** (Wave 5 also forged SLSA). Signed provenance is no longer a sufficient defense on its own.
- Payload: preinstall script, credential harvesting (npm, git, CI/CD tokens, **GCP and Azure cloud identities added** in this variant), self-spreading to other packages the victim can publish.
- Campaign marker: "Miasma: The Spreading Blight."
- **Resolved**: Red Hat closed the incident June 17. No Red Hat products confirmed built with compromised versions.
- Sources: [Wiz blog](https://www.wiz.io/blog/miasma-supply-chain-attack-targeting-redhat-npm-packages), [Microsoft Security Blog](https://www.microsoft.com/en-us/security/blog/2026/06/02/preinstall-persistence-inside-red-hat-npm-miasma-credential-stealing-campaign/), [Red Hat RHSB-2026-006](https://access.redhat.com/security/vulnerabilities/RHSB-2026-006)

**Action: MITIGATE NOW.** SLSA attestation forgery via OIDC token abuse is now a confirmed repeatable technique. The attack chain -- compromised developer account -> OIDC token -> forged SLSA -> published npm -- is exactly the path that would compromise `@nx/*` packages. Hardware-bound publish keys (YubiKey/HSM) + independent attestation verifier (not GitHub Actions-self-signed) are the required mitigations.

### HIGH: AsyncAPI npm Supply Chain -- Wave 7 "miasma-train-p1" (July 14, 2026)

- **5 @asyncapi packages** trojanized across 4 repos (`@asyncapi/specs` x2, `@asyncapi/generator`, `@asyncapi/generator-components`, `@asyncapi/generator-helpers`).
- Attack method: attacker pushed commits under a placeholder git identity; exploited each repo's existing release workflow via npm's GitHub OIDC trusted-publisher integration to mint short-lived tokens and publish. No developer account compromise needed.
- **Import-time payload delivery** (not postinstall hooks): executes at module-load (`require`/`import`) time -- **bypasses npm v12's postinstall-blocking defense directly**.
- Payload: Miasma RAT / botnet loader with 6 modules: credential harvest, encrypted exfiltration, supply-chain propagation, metamorphic generation, **AI-tool poisoning**, sandbox evasion. Modules 2-6 were embedded but **disabled** in this build (suggesting a staging/test wave).
- C2 resilience: Nostr, Ethereum, BitTorrent DHT, libp2p, and IPFS -- censorship-resistant command channels.
- Status: All 5 malicious versions unpublished from npm. Clean versions are now the dist-tag for each package.
- Sources: [Microsoft Security Blog: AsyncAPI analysis](https://www.microsoft.com/en-us/security/blog/2026/07/15/unpacking-asyncapi-npm-supply-chain-compromise-import-time-payload-delivery/), [StepSecurity](https://www.stepsecurity.io/blog/compromised-next-branch-pushes-malicious-asyncapi-generator-generator-helpers-and-generator-components-to-npm), [Hacker News](https://thehackernews.com/2026/07/compromised-asyncapi-npm-packages.html), [Socket.dev](https://socket.dev/blog/asyncapi-supply-chain-attack), [Wiz](https://www.wiz.io/blog/m-red-team-asyncapi-supply-chain-compromise-via-github-actions)

**Action: MITIGATE NOW.** Import-time execution is the successor attack vector after npm v12 blocks postinstall. Any Nx project using `@asyncapi/*` packages should pin to the clean dist-tags. More critically: the AI-tool-poisoning module was present but disabled. The next wave may enable it. Review whether Nx CLI, nx-mcp, or Self-Healing CI agents run in environments that could auto-require compromised packages.

### HIGH: npm v12 Ships July 8 -- Postinstall Blocked by Default

npm v12 ships the most significant security redesign in its 16-year history:
- **Blocks postinstall/install/preinstall scripts by default** (from all dependencies).
- **Blocks Git dependencies** (direct or transitive) unless `--allow-git`.
- **Blocks remote URL (https tarball) dependencies** unless `--allow-remote`.
- Migration: run `npm approve-scripts` to see which packages have scripts; allowlist trusted ones in `package.json`; commit.
- Motivated by: North Korean attacks, the year of Shai-Hulud supply chain attacks (455,000 malicious packages in 2025 alone).
- Sources: [TechTimes npm v12 coverage](https://www.techtimes.com/articles/319890/20260708/npm-v12-ships-this-month-blocking-install-scripts-that-enabled-year-supply-chain-attacks.htm), [Aikido.dev blog](https://www.aikido.dev/blog/npm-v12-block-postinstall), [GitHub changelog: breaking changes](https://github.blog/changelog/2026-06-09-upcoming-breaking-changes-for-npm-v12/)

**Action: EVALUATE + MITIGATE.** Nx CLI and `create-nx-workspace` both install packages as part of their core workflows. Any `npx create-nx-workspace` or `nx generate` flow that installs packages with postinstall scripts will break on npm v12 users. Audit which Nx-managed packages and generators rely on postinstall scripts. Publish migration guidance. Wave 7 already demonstrated that attackers are moving to import-time payloads to bypass this defense.

### HIGH: GuardFall -- Structural Shell Injection in 10 of 11 Open-Source AI Agents (June 30, carryover)

- Research from Adversa AI (June 30, 2026): 10 of 11 popular open-source AI coding agents (548,000 combined GitHub stars) have a structural shell injection bypass.
- Affected: Aider, Cline, Goose, Open Interpreter, OpenHands, opencode, Plandex, Roo-Code, SWE-agent, NousResearch Hermes. Only **Continue** is architecturally safe.
- Root cause: mismatch between how security filters inspect commands and how Bash interprets them. Denylist patterns fail structurally -- not patchable by adding more rules.
- Exploitation: pointing an affected agent at a malicious repo or package can trigger arbitrary shell command execution with the user's full account access.
- Sources: [Hacker News: GuardFall](https://thehackernews.com/2026/06/guardfall-exposes-open-source-ai-coding.html), [Security Affairs](https://securityaffairs.com/194546/ai/guardfall-flaw-hits-10-of-11-popular-open-source-ai-agents.html), [Adversa AI blog](https://www.waxell.ai/blog/guardfall-ai-coding-agent-shell-injection-governance-2026)

**Action: EVALUATE.** Closed-source tools (Claude Code, Cursor) were not in scope for this research. However, Nx CLI's `run-commands` executor and any Nx generator that spawns shell processes should be audited against the GuardFall bypass patterns. Self-Healing CI agents that exec shell commands are the highest-risk surface.

### MEDIUM: Claude Code GitHub Action -- Prompt Injection via Bot Accounts (patched v2.1.128, context)
