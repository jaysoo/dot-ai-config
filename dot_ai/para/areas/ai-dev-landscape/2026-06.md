## 1. MCP Ecosystem

### Spec Changes

**2026-07-28 Release Candidate (locked May 21, 2026)** -- largest revision since launch. Final spec ships July 28.

Key changes from 2025-11-25:
- **Stateless protocol core**: `initialize` handshake and `Mcp-Session-Id` removed. Servers are now truly stateless -- deploy behind any round-robin load balancer, no sticky sessions, no shared session store. New `Mcp-Method` and `Mcp-Name` headers for router-friendly traffic. `ttlMs` + `cacheScope` fields for client-side caching.
- **Extensions framework**: reverse-DNS identifiers, independent versioning, formal governance. New capabilities ship as opt-in extensions before (optionally) moving to spec.
- **Tasks moves to extension**: the experimental Tasks primitive from 2025-11-25 is replaced by a redesigned Tasks extension that works with stateless transport.
- **MCP Apps extension**: server-rendered HTML UIs in sandboxed iframes.
- **Authorization hardening**: stricter OAuth/OIDC alignment.
- **Full JSON Schema 2020-12** for tool schemas.
- **Breaking**: anyone matching error code `-32002` or using experimental Tasks API must migrate.
- Source: [MCP RC blog post](https://blog.modelcontextprotocol.io/posts/2026-07-28-release-candidate/)

**SDK validation window (W25 update):** Tier 1 Python and TypeScript SDKs are in the 10-week validation window (May 21 -- July 28). TypeScript SDK v2 expected Q3 2026. Python SDK v1.27.0 is the current production-ready 1.x release (RFC 8707 OAuth resource validation, StreamableHTTP idle timeout, TasksCallCapability backport).

### Security (NEW)

- **200,000 vulnerable MCP instances** disclosed across IDEs, internal tools, and cloud services (2026 disclosure). Tool poisoning benchmarked at 60%+ success rate across major LLM agents, 72% on some models. Source: [ITECS](https://itecsonline.com/post/mcp-tool-poisoning-enterprise-ai-agent-security-2026/), [Practical DevSecOps](https://www.practical-devsecops.com/mcp-security-vulnerabilities/)
- arXiv 2603.22489: "Model Context Protocol Threat Modeling and Analyzing Vulnerabilities to Prompt Injection with Tool Poisoning" -- formal threat model now in arXiv corpus. Source: [arXiv](https://arxiv.org/html/2603.22489v1)

### Adoption

- **Cursor 3.6 Auto-review**: allowlisted MCP calls run immediately; sandboxable calls run in sandbox; everything else routes to classifier subagent. First editor-native MCP sandboxing policy.
- **Claude Code plugin auto-loading (v2.1.157)**: plugins in `.claude/skills` directories load automatically; `claude plugin init <name>` scaffolds new plugins. Lowers MCP/plugin distribution friction for Claude Code users.
- **Managed Agents on Cloudflare**: Anthropic + Cloudflare co-launched hosting for Managed Agents. "Brain" (orchestrator) stays at Anthropic; "hands" (execution) run on Cloudflare. Source: [Cloudflare blog](https://blog.cloudflare.com/claude-managed-agents/)
- **GitHub Copilot SDK GA (June 2)**: custom MCP tools + MCP server connections now part of the embeddable agent SDK. `nx-mcp` can be registered as a custom tool inside apps built on the Copilot SDK -- a new distribution path.

### Nx Relevance: HIGH

- **Stateless protocol core is a meaningful architecture unlock for `nx-mcp`**: no sticky sessions means any cloud deployment pattern works. But it also means anyone running the old session-based pattern needs to update before July 28.
- **MCP Tasks-as-extension**: if any `nx-mcp` design assumed Tasks would be in the core spec, it needs to switch to the extension model. RC window closes July 28.
- **Tool-poisoning benchmark (60%+)** is still the `nx-mcp` audit blocker. Don't ship Nx Cloud MCP server until assessed against arXiv 2603.22489.
- **W25 addition: SDK validation window closes July 28 -- same deadline as the spec.** Tier 1 SDKs must ship support by then; server maintainers (including `nx-mcp`) have until that date to validate against the new stateless transport.

- **Cursor 3.8 Automations (W25)**: GitHub and Slack webhook triggers for always-on agents with computer use. Another entry in the "editor becomes CI trigger" pattern. freeCodeCamp course (June 15) on building self-healing CI pipelines confirms the pattern is now teachable/repeatable.

### HIGH: Miasma (Shai-Hulud Wave 6) -- Red Hat npm, Phantom Gyp (June 1-4, W25)

**June 1**: Attackers injected malicious code into at least 32 package releases under the @redhat-cloud-services npm namespace (Red Hat Hybrid Cloud Console frontend components + API clients). ~80,000 weekly downloads affected.

**June 3-4**: Same worm family returned with "Phantom Gyp" -- compromised 57 more packages via abused binding.gyp file.

**Technical innovation (Phantom Gyp):** A weaponized 157-byte binding.gyp triggers node-gyp to execute attacker-controlled code automatically during npm install -- without using preinstall or postinstall lifecycle scripts. Bypasses all install-script-based security monitoring.

**SLSA attestation forgery (again):** Attackers abused GitHub Actions' OpenID Connect trusted-publishing mechanism to mint short-lived npm tokens and publish maliciously modified packages bearing forged SLSA provenance attestations. Signed attestations could not block this pipeline hijack.

**Root cause**: A single compromised employee GitHub account. Credentials appeared in commercial infostealer logs as early as April 13, 2026 -- a 7-week exposure gap before exploitation.

**Payload**: Harvests developer and CI/CD credentials (npm, GitHub, AWS, GCP, Azure, HashiCorp Vault, Kubernetes). Exfiltrates via attacker-controlled GitHub repos. Injects GitHub Actions workflows for persistence. Self-propagates by republishing from any maintainer account it can reach.

Sources: [Wiz Miasma](https://www.wiz.io/blog/miasma-supply-chain-attack-targeting-redhat-npm-packages), [Snyk node-gyp](https://snyk.io/blog/node-gyp-supply-chain-compromise-self-propagating-npm-worm-binding-gyp/), [StepSecurity Phantom Gyp](https://www.stepsecurity.io/blog/binding-gyp-npm-supply-chain-attack-spreads-like-worm), [OX Security](https://www.ox.security/blog/600000-monthly-downloads-affected-miasma-supply-chain-attack-is-back-on-npm/), [JFrog](https://research.jfrog.com/post/shai-hulud-miasma-redhat-cloud-services/), [CSA research note](https://labs.cloudsecurityalliance.org/research/csa-research-note-miasma-npm-supply-chain-redhat-20260603-cs/), [TechTimes](https://www.techtimes.com/articles/317832/20260605/red-hat-npm-packages-compromised-57-more-follow-signed-attestations-cannot-block-pipeline-hijack.htm)

**Action: MITIGATE NOW (escalated).** Phantom Gyp confirms: preinstall/postinstall monitoring AND SLSA attestations can both be bypassed simultaneously. `@nx/*` security audit must now explicitly cover binding.gyp files. Hardware-bound keys are still necessary but not sufficient if the GitHub account used for publishing is compromised via infostealer. **Audit @nx/* publishing employee accounts against commercial infostealer datasets.** Red Hat had a 7-week gap -- this is the window Nx needs to close.
