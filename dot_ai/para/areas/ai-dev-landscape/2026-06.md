## 1. MCP Ecosystem

### Spec Changes

**2026-07-28 Release Candidate (locked May 21, 2026)** -- largest revision since launch. Final spec ships July 28.

Key changes from 2025-11-25:
- **Stateless protocol core**: `initialize` handshake and `Mcp-Session-Id` removed. Servers are now truly stateless -- deploy behind any round-robin load balancer, no sticky sessions, no shared session store. New `Mcp-Method` and `Mcp-Name` headers for router-friendly traffic. `ttlMs` + `cacheScope` fields for client-side caching.
- **Extensions framework**: reverse-DNS identifiers, independent versioning, formal governance. New capabilities ship as opt-in extensions before (optionally) moving to spec.
- **Tasks moves to extension**: the experimental Tasks primitive from 2025-11-25 is replaced by a redesigned Tasks extension that works with stateless transport.
- **MCP Apps extension**: server-rendered HTML UIs in sandboxed iframes.
- **Authorization hardening**: stricter OAuth/OIDC alignment.
- **Full JSON Schema 2020-12** for tool schemas.
- **Breaking**: anyone matching error code `-32002` or using experimental Tasks API must migrate.
- Source: [MCP RC blog post](https://blog.modelcontextprotocol.io/posts/2026-07-28-release-candidate/)

### Adoption (W24 update)

- **16,500+ indexed MCP servers** by June 2026 (PulseMCP growing ~1,000/month in Q1-Q2). Official MCP Registry API: 9,652 latest records, 28,959 server/version records as of May 24.
- **41% of surveyed software orgs in limited or broad production with MCP** (Stacklok 2026 software report). Transition from stdio to hosted transport and from API keys to OAuth 2.1 is mid-stream.
- Source: [MCP adoption statistics](https://www.digitalapplied.com/blog/mcp-adoption-statistics-2026-model-context-protocol), [The New Stack](https://thenewstack.io/model-context-protocol-roadmap-2026/)

### Security (June 1 baseline)

- **200,000 vulnerable MCP instances** disclosed across IDEs, internal tools, and cloud services (2026 disclosure). Tool poisoning benchmarked at 60%+ success rate across major LLM agents, 72% on some models. Source: [ITECS](https://itecsonline.com/post/mcp-tool-poisoning-enterprise-ai-agent-security-2026/), [Practical DevSecOps](https://www.practical-devsecops.com/mcp-security-vulnerabilities/)
- arXiv 2603.22489: "Model Context Protocol Threat Modeling and Analyzing Vulnerabilities to Prompt Injection with Tool Poisoning" -- formal threat model now in arXiv corpus. Source: [arXiv](https://arxiv.org/html/2603.22489v1)

### Adoption (June 1 baseline)

- **Cursor 3.6 Auto-review**: allowlisted MCP calls run immediately; sandboxable calls run in sandbox; everything else routes to classifier subagent. First editor-native MCP sandboxing policy.
- **Claude Code plugin auto-loading (v2.1.157)**: plugins in `.claude/skills` directories load automatically; `claude plugin init <name>` scaffolds new plugins. Lowers MCP/plugin distribution friction for Claude Code users.
- **Managed Agents on Cloudflare**: Anthropic + Cloudflare co-launched hosting for Managed Agents. "Brain" (orchestrator) stays at Anthropic; "hands" (execution) run on Cloudflare. Source: [Cloudflare blog](https://blog.cloudflare.com/claude-managed-agents/)

### Nx Relevance: HIGH

- **Stateless protocol core is a meaningful architecture unlock for `nx-mcp`**: no sticky sessions means any cloud deployment pattern works. But it also means anyone running the old session-based pattern needs to update before July 28.
- **MCP Tasks-as-extension**: if any `nx-mcp` design assumed Tasks would be in the core spec, it needs to switch to the extension model. RC window closes July 28.
- **Tool-poisoning benchmark (60%+)** is still the `nx-mcp` audit blocker. Don't ship Nx Cloud MCP server until assessed against arXiv 2603.22489.

### HIGH (NEW W24): Miasma -- CI Pipeline Hijack via GitHub Actions OIDC, SLSA Provenance Invalid (June 1-3)

- Wave 1 (June 1): 32 @redhat-cloud-services npm packages, 96 malicious versions. 116,991 weekly downloads affected.
- **Attack vector**: compromised Red Hat employee GitHub account -> orphan commit requests OIDC publishing token -> malicious packages published with **valid SLSA Build Level 3 provenance attestations**. The signing chain is intact; the pipeline generating it is compromised.
- Payload: npm preinstall hook drops 4.29 MB obfuscated credential stealer. Harvests GitHub/npm tokens, AWS/GCP/Azure/K8s credentials, HashiCorp Vault tokens, SSH keys, Docker registry credentials, GPG keys, .env files.
- Wave 2 (June 3, Phantom Gyp variant): 57 packages, 286+ malicious versions in under 2 hours. Largest victim: @vapi-ai/server-sdk (408,000+ monthly downloads). Used compromised binding.gyp / Trusted Publishing path.
- Sources: [Wiz Research](https://www.wiz.io/blog/miasma-supply-chain-attack-targeting-redhat-npm-packages), [StepSecurity (Phantom Gyp)](https://www.stepsecurity.io/blog/binding-gyp-npm-supply-chain-attack-spreads-like-worm), [Snyk](https://snyk.io/blog/miasma-supply-chain-attack-malicious-code-redhat-cloud-services-npm-packages/), [Microsoft Security](https://www.microsoft.com/en-us/security/blog/2026/06/02/preinstall-persistence-inside-red-hat-npm-miasma-credential-stealing-campaign/), [The Hacker News](https://thehackernews.com/2026/06/miasma-supply-chain-attack-compromises.html)
- **Action: MITIGATE NOW (revised from prior SLSA recommendation).** SLSA Build Level 3 is no longer sufficient as the sole supply-chain control. Required additions: (1) step-security/harden-runner on all npm-publishing Actions workflows; (2) no PR-triggered publish paths; (3) pin all GitHub Action versions with SHA; (4) require npm publish to happen only from a dedicated hardened runner, not shared CI. Treat any GitHub account with npm publish OIDC scope as a high-value target requiring hardware MFA.

- Discovered by JFrog researchers; contained before spreading to high-download packages.
- Sources: [BleepingComputer](https://www.bleepingcomputer.com/news/security/new-ironworm-malware-hits-36-packages-in-npm-supply-chain-attack/), [Dark Reading](https://www.darkreading.com/cyberattacks-data-breaches/rust-written-ironworm-npm-supply-chain), [Phoenix Security](https://phoenix.security/ironworm-npm-supply-chain-worm-rust-ebpf-rootkit-tor/), [OX Security](https://www.ox.security/blog/ironworm-supply-chain-malware-hits-npm/)

Audit which Nx Cloud and self-hosting environments have `ANTHROPIC_API_KEY` set and confirm use of short-lived scoped keys. The eBPF rootkit means standard container isolation is insufficient -- assume any compromised developer machine is a persistent threat even after key rotation.
