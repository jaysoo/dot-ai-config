### Risks for Nx

- **Security CVEs (June 18 batch)**: HIGH-severity TLS and WebCrypto vulnerabilities affect all supported lines. Verify Nx CI matrix is running patched versions (v22.23.x / v24.17.x+ / v26.3.1+).
- **HTTP/2 priority signaling removed in Node 22.23.0**: Backport of a SEMVER-MAJOR to Maintenance LTS. Any Nx or Nx Cloud code using HTTP/2 with priority frames needs audit.
- **`Date` properties on `Stats` now enumerable (Node 24.18.0)**: Code that spreads or iterates Stats objects will now see `atime`, `mtime`, `ctime`, `birthtime` as enumerable. Unlikely to break Nx but grep `...stats` patterns.
- **`node:vfs` (Node 26.4.0)**: If Nx plugins mock fs for testing, check for conflicts with the new VFS subsystem. Also evaluate if it simplifies Nx's own virtual-fs scenarios.
- **Package maps for import resolution (Node 26.4.0)**: May interact with Nx's module resolution overrides and path mappings in plugin loading.
- **Carryover**: `module.register()` runtime-deprecated (DEP0205), `_stream_*` modules removed, `Temporal` on Stats -- all from prior scans, still unverified against Nx codebase.

### Opportunities

- **`node:vfs` (Node 26.4.0)**: Potential for cleaner Nx plugin sandboxing and virtual project graph without custom fs-mock layers.
- **Buffer poolSize 64KiB (Node 24.18.0)**: Large Nx project graphs that do heavy buffer operations may see memory efficiency gains -- no code change needed, just a Node version pin.
- **New crypto algorithms (Node 24.18.0 + 26.2.0)**: ML-DSA, ML-KEM, ChaCha20-Poly1305. Relevant if Nx Cloud artifact signing or cache integrity checks add post-quantum options.
- **`closeIdleConnections()` (Node 26.4.0)**: Nx Cloud daemon keeps persistent HTTP connections. This API allows graceful teardown without tracking sockets manually.
- **Carryover**: `randomUUIDv7()`, `require(esm)`, module compile cache, `stream.compose` stable.

### Releases since 2026-06-01

| Version | Date | Key Changes |
|---------|------|-------------|
| **v2.8.2** | 2026-06-03 | **Parallel workspace task execution** (recursive tasks), `--bundle` flag for `deno compile`, npm native binary execution, improved CJS re-exports, DNS lookup error reporting, `node:vm` dynamic imports, scoped registry auth for same-origin tarballs |
| **v2.8.3** | 2026-06-11 | **Workspace auto-discovery** of external deno.json import maps, **lifecycle scripts for workspace packages**, workspace deps with aliases in package.json, Node.js v24.15.0 emulation, `deno add` accepts npm version ranges, ML-DSA/ML-KEM JWK crypto |
| **v2.9.0** | 2026-06-25 | **Workspace node_modules symlink** (workspace npm members into root node_modules by name), **lockfile migration** from pnpm-workspace.yaml / package-lock.json / yarn.lock / pnpm-lock.yaml / bun.lock -> deno.lock, JSR deps via npm-compat registry into node_modules, Node.js v26.3.0 emulation (Node-API v10), Web Locks API, breaking: Deno.serve compression requires explicit opt-in |

### Highlights (v2.9.0)

- **Lockfile migration from pnpm/yarn/bun**: `deno` can now seed `deno.lock` from existing ecosystem lockfiles. This means a team can onboard Deno tooling into an existing pnpm monorepo without a full lockfile rewrite. Directly reduces the switching cost from npm-ecosystem tools.
- **Workspace node_modules symlinking**: Workspace packages appear in root `node_modules` by name, the same pattern npm/pnpm use. This resolves a major ergonomic gap for Deno monorepos.
- **Node.js v26.3.0 emulation**: Deno is now emulating the latest Node patch, not trailing a major version behind.
- **Parallel workspace tasks (v2.8.2)**: Deno now runs recursive workspace tasks in parallel. Directly overlaps with `nx run-many --parallel`.
- **Lifecycle scripts (v2.8.3)**: `postinstall`, `prepare` etc. now run in workspace packages. One more pnpm feature Deno now matches.

### Nx compat status

- **Competitive signal elevated**: Deno 2.9.0 is the biggest threat to pnpm/Nx monorepo workflows in the 2026 tracking window. The combination of lockfile migration + workspace node_modules + parallel tasks + lifecycle scripts means a Deno-first team can replicate most pnpm monorepo basics.
- **Nx-on-Deno**: Node compat at v26.3.0 emulation. Most Nx CLI operations that don't touch workspace-specific patching should work. Plugin loaders that use `module.register()` will hit the same runtime deprecation as on real Node 26.
- **Gap remaining**: Nx's task orchestration (affected graph, distributed execution, cache) is still far ahead of Deno's parallel task runner. But the gap at the dependency management layer has narrowed significantly.

**Risk level for Nx: LOW-MEDIUM -> MEDIUM** (nudge up: 2.9.0 workspace node_modules + lockfile migration is a real inflection point).
