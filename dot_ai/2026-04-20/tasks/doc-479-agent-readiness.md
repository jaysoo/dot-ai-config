# DOC-479: Low-Effort Agent-Readiness Improvements for nx.dev

**Linear:** https://linear.app/nxdev/issue/DOC-479/review-low-effort-agent-readiness-improvements-for-nxdev
**PRs:** #35348 (merged), #35351 (sitemap follow-up, working on preview)

## Context

isitagentready.com scan flagged two gaps on nx.dev: no agent-useful Link
relation types in response headers, and no Content-Signal directives in
robots.txt. Jack also asked for preliminary research on Agent Skills index.

## Shipped (#35348)

1. **Link response headers (RFC 8288)** on Framer-proxied pages ‚Äî added to
   `netlify/edge-functions/rewrite-framer-urls.ts`:
   ```
   Link: </llms.txt>; rel="describedby", </llms-full.txt>; rel="service-doc",
         </sitemap.xml>; rel="sitemap"
   ```
   Applies to homepage + marketing pages.

2. **Content-Signal in robots.txt** ‚Äî added permissive directive in
   `astro-docs/src/pages/robots.txt.ts`:
   ```
   Content-Signal: search=yes, ai-input=yes, ai-train=yes
   ```
   Permissive because nx.dev actively wants AI agents to consume docs
   (ships MCP server, llms.txt, etc.).

3. **`nx.dev/robots.txt` now served from astro-docs** ‚Äî added beforeFiles
   rewrite in `nx-dev/nx-dev/next.config.js`:
   ```js
   { source: '/robots.txt', destination: `${astroDocsUrl}/docs/robots.txt` }
   ```
   Used `beforeFiles` (not default `afterFiles`) to win over any generated
   `public/robots.txt`. Also fixed pre-existing typo: robots.txt sitemap URL
   was `/sitemap-index.xml` (404) ‚Üí corrected to `/sitemap.xml`.

## Deferred

**Agent Skills Discovery index** (per Cloudflare RFC v0.2.0) ‚Äî researched
thoroughly. Recommendation: **defer**. Evidence of absence, not just
absence of evidence:
- RFC author admits "no major public deployments" in [issue #4](https://github.com/cloudflare/agent-skills-discovery-rfc/issues/4)
- agentskills.io's ~35 listed clients consume SKILL.md from **local filesystem**, not HTTP
- GitHub code search `/.well-known/agent-skills/index.json`: ~22 hits, ~20 publisher-side, only ~4 hobbyist consumer-side scanners
- Zero Cloudflare products consume it; no Claude Code / Cursor / Copilot / Codex release notes mention web-hosted skill index discovery

If/when a real consumer ships, implement minimal static version listing the
3-4 flat skills from `nrwl/nx-ai-agents-config`. ~1 hour.

## Post-merge debugging

After #35348 merged, `nx.dev/robots.txt` still served Framer's URL. Three
hops to root cause:

1. Suspected stale CDN cache ‚Äî wrong, the prod deploy artifact literally had
   `public/robots.txt` baked in.
2. Thought `public/robots.txt` was a gitignore-only ghost ‚Äî wrong, it was
   being generated by `next-sitemap.config.js` via `generateRobotsTxt: true`.
3. That config was **deleted in DOC-478 (#35315)**, but prod was still
   pinned to an older deploy whose build ran pre-cleanup.

Fresh deploy from latest master ‚Üí `beforeFiles` rewrite fires ‚Üí astro-docs
robots.txt with Content-Signal served. ‚úì

## Sitemap regression (#35351)

DOC-478 cleanup also removed `/sitemap.xml` generation ‚Äî but unlike
`/robots.txt`, there was no rewrite fallback. So `nx.dev/sitemap.xml` 404'd
post-cleanup, which broke SEO.

**Fix**: restored minimal sitemap infrastructure on `fix_nxdev_sitemap`:
- `nx-dev/nx-dev/next-sitemap.config.js` with `generateRobotsTxt: false`
  (so the astro-docs robots.txt rewrite still wins) and
  `exclude: noIndex ? ['/*'] : ['/ai-chat', '/api/*']`
- `nx-dev/nx-dev/scripts/patch-sitemap-index.mjs` (unchanged ‚Äî appends
  `/sitemap-1.xml` and `/docs/sitemap-index.xml` to the index)
- `nx-dev/nx-dev/project.json` ‚Äî re-added `sitemap` target, `build.dependsOn`
  includes `sitemap`, `next:build.outputs` includes `public/sitemap*.xml`

Local verification (`nx run nx-dev:sitemap --skip-nx-cache`):
- `sitemap.xml` index: 3 entries (sitemap-0 + sitemap-1 + docs/sitemap-index) ‚úì
- `sitemap-0.xml`: 23 URLs (`/courses`, course roots, lessons) ‚úì

Preview appears empty because `NEXT_PUBLIC_NO_INDEX=true` on preview context
‚Äî correct behavior for a noindex deploy, cache-safe because the env var is
in the target's input hash.

## Lessons

- Don't trust `.gitignore` as evidence a file is generated ‚Äî check build
  scripts too. I initially assumed a `next-sitemap` invocation existed in
  the repo; it was Netlify-dashboard-invisible (turned out to just be the
  config file triggering it via pnpm CLI).
- Netlify Edge CDN aggressively caches static assets regardless of
  `max-age=0` ‚Äî redeploys don't always bust it; need "Clear cache and
  deploy" or rebuild fresh.
- next-sitemap v3.1.10 **does** auto-detect App Router routes ‚Äî my initial
  `additionalPaths` enumeration was redundant and produced wrong-format URLs
  (missing the `lessons-` prefix added by `courses.api.ts:103`).
